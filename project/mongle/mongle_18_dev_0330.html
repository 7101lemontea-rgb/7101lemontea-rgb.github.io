Overview

작업 개요

Mongle-app(Expo Router 기반 React Native 웨딩 플래닝 앱) 개발 과정에서 수행한 세 가지 주요 작업을 통합 정리한 개발 작업 문서입니다.

Phase A 오류 해결

모듈 경로, 중복 스크린, 탭바 렌더링 등 라우팅 문제 수정

Phase B 채팅 다중 초대

chat_members 테이블 추가, 초대 코드 · 멤버 관리 구현

Phase C Supabase 연동

RLS 정책, 마이그레이션, 트러블슈팅 내역 정리

01 — Bug Fixes

오류 해결 내역

1-1. 모듈 경로 오류 (Unable to resolve)

Unable to resolve "../../../hooks/useAi" from "app\(couple)\(tabs)\chat\ai.jsx"

발생 위치 ai.jsx, dashboard.jsx 등 다수 파일
원인
Expo Router 중첩 라우팅 구조에서 상대경로 계산 오류. ../../../는 app/까지만 이동하여 프로젝트 루트에 도달하지 못함.
해결
@/ 절대경로로 통일 (근본 해결)
수정 전 ❌ 
import { useAi } from '../../../hooks/useAi'
수정 후 ✅ 
import { useAi } from '@/hooks/useAi'

1-2. 중복 Screen 오류 (Duplicate screen named "chat")

Error: A navigator cannot contain multiple Screen components with the same name (found duplicate screen named "chat")

원인 동일한 "chat" 스크린 경로 중복 존재
원인
app/(planner)/(tabs)/chat/와 app/(planner)/chat/가 동시에 존재하여 Expo Router가 동일한 "chat" 스크린으로 인식, 충돌 발생.
해결
app/(planner)/chat/ 폴더 전체 삭제. app/(planner)/(tabs)/chat/만 유지.

1-3. 탭바 2중 렌더링 및 불필요한 탭 노출

원인 Screen 선언명과 실제 파일명 불일치 + 숨김 처리 누락
증상
하단 탭 내비게이션이 2개 겹쳐서 표시됨. index, docs, budget, (tabs), customer 등 숨겨야 할 탭이 노출됨.
해결
존재하지 않는 Screen 선언 제거 후, 탭바에 노출되면 안 되는 실제 파일에 href: null 추가. 
// (planner)/_layout.jsx — 탭바 숨김 처리 추가
<Tabs.Screen name="budget"        options={{ href: null }} />
<Tabs.Screen name="docs"          options={{ href: null }} />
<Tabs.Screen name="index"         options={{ href: null }} />
<Tabs.Screen name="customer/[id]" options={{ href: null }} />

1-4. Layout children 경고 (No route named "X" exists)

[Layout children]: No route named "customer" exists in nested children: [..., "customer/[id]"]

원인 폴더명과 Screen name 불일치
원인
_layout.jsx에서 name="customer"로 선언했으나 실제 파일은 customer/[id].jsx만 존재. customer/index.jsx 파일 없음.
수정 전 ❌ 
<Tabs.Screen name="customer"      options={{ href: null }} />
수정 후 ✅ 
<Tabs.Screen name="customer/[id]" options={{ href: null }} />
02 — Folder Structure

폴더 구조 개선

vendors 폴더 접근 문제

(tabs) 폴더 신규 생성 후 바깥의 vendors 폴더에 접근할 수 없어 업체 리스트 클릭 시 업체 정보 화면으로 이동되지 않는 문제가 발생했습니다. vendors/ 폴더를 (tabs)/ 안으로 이동하고 _layout.jsx에 vendors/[id] 숨김 처리를 추가하여 해결했습니다.

변경 전 ❌ 
app/(planner)/
├── (tabs)/
│   └── ...
└── vendors/
    └── [id].jsx
변경 후 ✅ 
app/(planner)/
├── (tabs)/
│   ├── vendors/
│   │   └── [id].jsx
│   └── ...
└── _layout.jsx
03 — Git

Git 작업 내역

로컬 변경사항과 원격 저장소 충돌 발생 시, 로컬 변경사항을 폐기하고 원격 코드를 Pull하는 방법입니다. 

# 로컬 변경사항 전부 폐기 후 원격 코드 pull
git reset --hard HEAD && git pull

⚠️ 이 명령어는 저장되지 않은 로컬 변경사항을 전부 삭제합니다. 실행 전 반드시 보존할 내용이 없는지 확인하세요.

04 — DB Schema

DB 스키마

기존 projects (user_id 1개) → chats → messages 구조에서, chats → chat_members (여러 user_id, 초대로 추가) → messages 구조로 변경되어 다중 사용자 채팅이 가능해졌습니다.

chats

컬럼명 타입 비고
id PK uuid NOT NULL
project_id FK uuid NOT NULL
title text NULL
invite_code text UNIQUE — 신규 추가
created_at timestamptz now()
updated_at timestamptz now()

chat_members 신규 추가

컬럼명 타입 비고
id PK uuid NOT NULL
chat_id FK uuid CASCADE
user_id FK uuid CASCADE
role text owner / member
joined_at timestamptz now()

messages

컬럼명 타입 비고
id PK uuid NOT NULL
chat_id FK uuid NOT NULL
sender text NOT NULL
content text NOT NULL
created_at timestamptz now()

profiles

컬럼명 타입 비고
id PK uuid NOT NULL
role text NULL
planner_id uuid NULL
couple_id uuid NULL
expo_push_token text NULL
05 — RLS Policies

RLS 보안 정책

5-1. chats 테이블

정책명 명령 조건 (USING)
Users can manage own chats ALL EXISTS (SELECT 1 FROM projects WHERE projects.id = chats.project_id AND projects.user_id = auth.uid())
members can view chats SELECT auth.uid() IN (SELECT chat_members.user_id FROM chat_members WHERE chat_members.chat_id = chats.id)
초대 코드로 채팅방 조회 SELECT invite_code IS NOT NULL

'초대 코드로 채팅방 조회' 정책을 추가하여 미가입 멤버도 초대 코드로 채팅방 조회가 가능하도록 수정했습니다.

5-2. chat_members 테이블

정책명 명령 조건
멤버 조회 SELECT auth.uid() = user_id
본인만 멤버로 추가 INSERT auth.uid() = user_id
멤버 삭제 DELETE auth.uid() = user_id

⚠️ RLS 정책에서 chat_members를 다시 SELECT하는 구조는 무한 재귀(infinite recursion)를 유발합니다. 반드시 auth.uid() = user_id 방식으로 단순하게 작성해야 합니다.

06 — Chat Feature

채팅 기능 상세

6-1. 초대 코드 기반 채팅방 참여 (index.jsx)

코드로 입장
  • FAB 위에 열쇠 아이콘 버튼(joinBtn) 추가
  • 초대 코드 입력 모달에서 코드 입력 → chats 테이블에서 invite_code 일치하는 방 조회
  • 이미 참여 중인 경우 중복 없이 해당 채팅방으로 바로 이동
  • 신규 참여 시 chat_members에 role: 'member'로 insert
초대 코드 보기
  • 채팅방 롱프레스 → '초대 코드 보기' 메뉴 선택
  • chats 테이블에서 invite_code 조회 후 큰 글씨로 모달 표시
  • 클립보드 복사 버튼(expo-clipboard) 제공

6-2. 멤버 초대 및 관리 ([id].jsx)

헤더 점 3개 메뉴
방치되어 있던 menuBtn에 setMenuVisible(true) 연결. 메뉴 시트: 채팅방 제목, 멤버 수, 멤버 목록, 멤버 초대 버튼, 닫기 버튼 표시.
초대 코드 생성 및 공유
  • '코드 생성' 버튼: 6자리 랜덤 코드 생성 후 chats.invite_code 업데이트
  • '코드 재생성' 버튼: 기존 코드 무효화 후 새 코드로 교체
  • '공유' 버튼: Share.share()로 OS 기본 공유 시트 호출
멤버 강퇴 방장 전용
isOwner가 true인 경우에만 멤버 행에 강퇴 버튼 표시. 강퇴 시 chat_members에서 해당 user_id 삭제 + 시스템 메시지 전송.

6-3. 채팅방 생성 흐름 개선 (index.jsx)

  • 생성자를 chat_members에 role: 'owner'로 자동 등록
  • project_id 없는 신규 유저는 '내 웨딩 플랜' 기본 프로젝트 자동 생성
  • insert 시 state 변수 projectId 대신 로컬 변수 pid 사용으로 타이밍 버그 수정
07 — Issues

이슈 및 해결 내역

Issue 01 초대 코드 입력 시 '유효하지 않은 초대 코드' 오류
원인
chats RLS가 기존 멤버/소유자만 SELECT 허용하여 미가입자의 invite_code 조회가 0건 반환됨.
해결
invite_code IS NOT NULL 조건의 SELECT 정책 추가.
Issue 02 이메일 초대 시 '해당 이메일이 없다' 오류
원인
profiles 테이블에 email 컬럼 없어 이메일로 유저 조회 불가.
해결
이메일 초대 방식 제거, 초대 코드 방식으로 전환.
Issue 03 헤더 점 3개 버튼 클릭 시 아무 동작 없음
원인
[id].jsx menuBtn에 onPress 핸들러 미연결.
해결
onPress={() => setMenuVisible(true)} 연결.
Issue 04 채팅방 목록/내부 화면 무한 로딩
원인
try-catch-finally 미적용으로 에러 시 setLoading(false) 미호출.
해결
모든 비동기 함수에 finally 블록 추가.
Issue 05 chat_members와 profiles 조인 오류
원인
Supabase 조인은 외래키 관계 필요.
해결
분리 조회 후 JS에서 병합 처리.
Issue 06 RLS 무한 재귀 오류
원인
정책에서 chat_members를 재귀 참조.
해결
auth.uid() = user_id 단순 비교 방식으로 교체.

09 — SQL

SQL 실행 내역

9-1. chat_members 테이블 생성

CREATE TABLE chat_members (
  id        uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  chat_id   uuid NOT NULL REFERENCES chats(id) ON DELETE CASCADE,
  user_id   uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  role      text,
  joined_at timestamptz DEFAULT now(),
  UNIQUE (chat_id, user_id)
);
ALTER TABLE chat_members ENABLE ROW LEVEL SECURITY;

9-2. invite_code 컬럼 추가

ALTER TABLE chats ADD COLUMN invite_code TEXT UNIQUE;

9-3. RLS 정책 추가

-- chats: 초대 코드 조회 정책
CREATE POLICY "초대 코드로 채팅방 조회"
  ON chats FOR SELECT
  USING (invite_code IS NOT NULL);

-- chat_members: 본인만 추가 가능
CREATE POLICY "본인만 멤버로 추가"
  ON chat_members FOR INSERT
  WITH CHECK (auth.uid() = user_id);

-- chat_members: 본인 행만 조회
CREATE POLICY "멤버 조회"
  ON chat_members FOR SELECT
  USING (auth.uid() = user_id);

9-4. 기존 채팅방 데이터 마이그레이션

코드 적용 이전에 생성된 채팅방은 chat_members에 데이터가 없으므로 수동으로 등록해야 합니다. 

INSERT INTO chat_members (chat_id, user_id, role)
VALUES
  ('<chat_id>', '<user_id>', 'owner');
-- 기존 채팅방마다 소유자 user_id를 확인하여 반복 실행
10 — Recommendations

향후 권장 사항

개발 환경 / 코드 품질

  • 모든 import에 @/ 절대경로 사용으로 통일하여 경로 오류 방지
  • 새 파일/폴더 추가 시 _layout.jsx의 Screen 선언과 동기화 필수
  • 파일 추가/삭제 후 npx expo start --clear로 캐시 초기화 습관화
  • 폴더 구조 변경 시 라우팅 경로(router.push) 일괄 점검 필요

기능 고도화

Todo 01 초대 코드 만료 시간 추가
expired_at 컬럼 추가로 보안 강화.
Todo 02 채팅방 나가기 시 방장 위임 기능
현재는 방장 퇴장 시 채팅방이 삭제됨. 방장 위임 기능 구현 필요.
Todo 03 읽지 않은 메시지 수 뱃지 표시
messages 테이블에 read_at 컬럼 추가.
Todo 04 알림 음소거 상태 DB 저장
chat_members.is_muted 컬럼으로 DB에 저장.
Todo 05 멤버 목록 이름 표시
profiles 테이블에 nickname 또는 display_name 컬럼 추가 시 멤버 목록에 이름 표시 가능.
Todo 06 프로젝트 생성 온보딩 플로우 구현
온보딩 플로우 구현 후 자동 생성 로직('내 웨딩 플랜') 제거 필요.
Todo 07 profiles ↔ chat_members 외래키 추가
외래키 추가 시 분리 조회 → 조인 방식으로 쿼리 최적화 가능.
