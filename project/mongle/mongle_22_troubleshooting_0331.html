ISSUE-01   웨딩플래너 "더보기" 버튼 클릭 시 화면 이동 안 됨

대상 파일: app/_layout.jsx  ·  증상: 더보기 버튼을 눌러도 화면이 이동하지 않고 홈 화면에 머물러 있음.

원인 파악 _layout.jsx 라우팅 가드 — planners 예외 처리 누락
경로 확인
app/planners/ 폴더 존재, /planners 경로 자체는 올바름.
근본 원인
(couple) 그룹 밖 경로로 이동 시 무조건 /(couple)로 강제 리다이렉트. planners 경로도 이 조건에 포함되어 이동 즉시 홈으로 복귀.
수정 전 ❌ 
const inSettingsGroup = rootSegment === 'settings';
if (!session) {
  if (!inAuthGroup && !inCoupleGroup) {
    router.replace('/(couple)'); // planners도 해당됨
  }
} else {
  } else if (!inCoupleGroup) {
    router.replace('/(couple)'); // planners도 해당됨
  }
}
수정 후 ✅ 
const inSettingsGroup = rootSegment === 'settings';
const inPlannerPage = rootSegment === 'planners'; // ✅ 추가
if (!session) {
  if (!inAuthGroup && !inCoupleGroup && !inPlannerPage) { // ✅
    router.replace('/(couple)');
  }
} else {
  } else if (!inCoupleGroup && !inPlannerPage) { // ✅
    router.replace('/(couple)');
  }
}

ISSUE-02   플래너 상세 페이지 조회 실패 (PGRST116)

대상 파일: app/(couple)/planners/[id].jsx  ·  증상: 플래너 카드 클릭 시 상세 페이지 미열림, "플래너를 찾을 수 없습니다" 에러

LOG error: {"code": "PGRST116", "details": "The result contains 0 rows"}

원인 역방향 Join으로 빈 배열 반환
파악 과정
id = 'planner1' 정상 전달 확인. mapPlanner에서 id: p.name으로 매핑 → 카드 클릭 시 /planners/planner1 이동. wedding_planners 테이블에 name='planner1' 행 존재 → .eq('name', id)는 올바른 방식.
근본 원인
user_profiles!planner_id Join 방향 오류. 외래키가 user_profiles.planner_id에 있으므로 역방향 Join이 올바르게 동작하지 않아 빈 배열 반환.
해결
Join 제거 후 .select('*')으로 단순 조회. 수신자 ID 조회는 handleConsultSubmit에서 별도 처리.
수정 전 ❌ 
// 역방향 Join 시도
const { data, error } = await supabase
  .from('wedding_planners')
  .select(`*, user_profiles!planner_id (id)`)
  .eq('name', id)
  .single();
수정 후 ✅ 
// Join 제거, 단순 조회
const { data, error } = await supabase
  .from('wedding_planners')
  .select('*')
  .eq('name', id)
  .single();
// 수신자 ID는 handleConsultSubmit에서 별도 처리

✅ 결과: 플래너 상세 페이지 정상 로드 확인

ISSUE-03   상담 신청 실패 — 사용자 정보를 식별할 수 없습니다

대상 파일: app/(couple)/planners/[id].jsx — handleConsultSubmit  ·  증상: "상담 신청하기" 클릭 시 "사용자 정보를 식별할 수 없습니다" 에러

1단계 user_profiles 결과 빈 배열 반환
Join 쿼리 방식 문제 → ISSUE-02에서 해결.
2단계 별도 쿼리 변경 후에도 0 rows 반환 
const { data: userProfile } = await supabase
  .from('user_profiles')
  .select('id')
  .eq('planner_id', selectedPlanner.id)
  .single();
// 결과: userProfile = null, error: PGRST116
3단계 RLS 정책 문제 확정
확인 방법
SQL Editor에서 직접 실행 → 정상 결과. 앱에서는 0 rows → RLS 차단 확인.
원인
couple 유저가 planner의 user_profiles 행을 SELECT할 권한 없음.
수정 1 RLS 추가 
CREATE POLICY "Anyone can read planner profiles"

  USING (role = 'planner');

}
await sendNotificationToPlanner({

#04   비용 현황 전부 0으로 표시

대상 파일: planner_budget.jsx  ·  증상: 비용 현황 화면의 수금 총액, 미수금이 전부 0으로 표시됨

원인 couple_payments 테이블 미조회
원인
couples.total_amount, received_amount가 DB에 0으로 저장. 실제 수금 데이터는 couple_payments 테이블에 있으나 해당 테이블을 조회하지 않음.
해결
couple_payments를 join 조회 후 집계. total_amount > 0이면 기존 값, 아니면 payments 합계로 동적 계산. 
const payments = c.couple_payments ?? [];
const totalFromPayments    = payments.reduce((s, p) => s + p.amount, 0);
const receivedFromPayments = payments
  .filter(p => p.paid)
  .reduce((s, p) => s + p.amount, 0);
return {
  total:    c.total_amount    > 0 ? c.total_amount    : totalFromPayments,
  received: c.received_amount > 0 ? c.received_amount : receivedFromPayments,
};

#05   비용 현황 전부 0으로 표시

대상 파일: dashboard.jsx  ·  증상: 대시보드 KPI 카드의 담당 커플 수가 0으로 표시됨

원인 couples.planner_id가 null — AuthContext에서 planner_id null 반환
원인
AuthContext에서 planner_id가 null 반환. .eq('planner_id', null) 조건에 매칭되는 row 없음.
해결
Supabase 대시보드에서 직접 수정. couples 테이블 각 row의 planner_id에 플래너 ID 입력. user_profiles 테이블 플래너 계정 row의 planner_id 입력.

확인 방법: 두 값이 일치해야 데이터 표시됨 — couples.planner_id = user_profiles.planner_id = auth.uid()

Chapter 03 — Auth · Context · Component Rendering

인증·Context·컴포넌트 렌더링 이슈

#01   Property 'profile' doesn't exist

대상 파일: app/_layout.jsx  ·  증상: AuthContext.Provider의 value에서 profile?.planner_id 접근 시 런타임 에러

원인 지역 변수를 컴포넌트 스코프에서 참조
원인
fetchUserRole() 반환값을 useEffect 내부 지역 변수로만 선언. Provider value가 있는 컴포넌트 스코프에서 profile 접근 불가.
해결
profile을 useState로 상태 승격. initialize()에서 setProfile(data)로 상태에 저장.
수정 전 ❌ 
const initialize = async () => {
  const profile = await fetchUserRole(session.user.id);
  setRole(profile?.role || null);
  // profile은 이 블록 안에서만 유효
};
// 에러: profile이 이 스코프에 없음
<AuthContext.Provider value={{ planner_id: profile?.planner_id }}>
수정 후 ✅ 
const [profile, setProfile] = useState(null); // 상태로 승격
const initialize = async () => {
  const data = await fetchUserRole(session.user.id);
  setProfile(data); // 상태에 저장
  setRole(data?.role || null);
};
// 정상 동작
<AuthContext.Provider value={{ planner_id: profile?.planner_id ?? null }}>

#02   planner_id 하드코딩 문제

대상 파일: dashboard.jsx  ·  증상: 플래너가 여러 명일 때 항상 동일한 플래너 데이터만 표시됨

수정 전 ❌ 
const PLANNER_ID = 'ab979ad2-5736-4dbe-8bee-336adef8e5e2';
const { data } = await supabase
  .from('couples').eq('planner_id', PLANNER_ID);
수정 후 ✅ 
const { planner_id } = useAuth(); // 동적 조회
const { data } = await supabase
  .from('couples').eq('planner_id', planner_id);

원칙: planner_id 하드코딩 절대 금지 — 반드시 useAuth()에서 동적으로 사용. useEffect 내 if (!planner_id) return; early return 필수.

#03   plannerId vs planner_id 네이밍 불일치

대상 파일: context/AuthContext.js  ·  증상: AuthContext에서 planner_id를 꺼내도 항상 undefined

구분 잘못된 값 올바른 값
Context 필드명 plannerId (camelCase) planner_id (snake_case)
원칙 camelCase는 DB와 불일치 DB 컬럼명(snake_case)과 동일하게 유지

#06   Identifier 'searchText' has already been declared

대상 파일: couple_list.jsx  ·  증상: SyntaxError: Identifier 'searchText' has already been declared

원인 새 state 블록 삽입 시 기존 선언 미삭제
원인
컴포넌트 상단에 새 state 블록 삽입 시 기존 선언을 삭제하지 않음. searchText, activeStage, addVisible 등 약 10개 useState가 두 번 선언됨.
해결
240~259번 줄의 중복 state 선언 블록 전체 삭제.

예방: 기존 파일 수정 시 새 블록 삽입 전 반드시 기존 선언 유무를 먼저 확인.

#07   ReferenceError: Property 'coupleName' doesn't exist

대상 파일: planner_todo_list.jsx  ·  증상: 할 일 목록 화면에서 coupleName 함수 접근 불가 에러

원인 클로저 범위 밖 함수 참조
원인
coupleName 함수가 PlannerTodoList 컴포넌트 내부에 정의. TodoItem은 컴포넌트 외부 선언 — JavaScript 클로저 범위 밖에서 참조 불가.
수정 전 ❌ 
<TodoItem item={t} ... />
// TodoItem 내부에서 직접 참조 — 에러
<Text>{coupleName(item)}</Text>
수정 후 ✅ 
// 호출부 — 미리 계산해 prop으로 전달
<TodoItem item={t} coupleName={coupleName(t)} ... />
// TodoItem 정의부
function TodoItem({ item, coupleName, isLast, onToggle, onEdit }) {
  <Text>{coupleName}</Text> // 정상 동작
}

#08   완료 토글 시 취소선이 표시되지 않음

대상 파일: dashboard.jsx  ·  증상: 항목 탭 시 완료 처리는 되지만 취소선 없이 항목이 바로 사라짐

원인 완료 토글 즉시 필터에서 항목 제거
원인
dashboardTodos를 매 렌더마다 urgentTodos + upcomingTodosFiltered에서 재계산. 두 목록 모두 !t.done 필터 적용 → 완료 토글 즉시 항목 제거. 취소선이 표시되기 전에 항목 자체가 사라지는 현상.
해결
useRef(lockedTodoIds)로 최초 로드 시 5개 ID 고정. 렌더 시 고정 ID 기준으로 최신 done 상태만 반영. 
// fetchTodos 최초 실행 시에만 ID 고정
const lockedTodoIds = useRef([]);
if (lockedTodoIds.current.length === 0) {
  lockedTodoIds.current = [...urgent, ...upcoming]
    .slice(0, 5).map(t => t.id);
}
// 렌더 시 고정 ID 기준으로 최신 done 상태 반영
const dashboardTodos = lockedTodoIds.current
  .map(id => todos.find(t => t.id === id))
  .filter(Boolean);

#09   TypeError: Cannot read property 'bg' of undefined

대상 파일: couple_list.jsx  ·  증상: 커플 카드 탭 또는 커플 추가 후 TypeError 발생

원인 1 avatarIdx 없음
DB에서 가져온 커플 데이터에 avatarIdx 컬럼 없음. undefined % 6 = NaN → AVATAR_POOL[NaN] = undefined → .bg 에러.
원인 2 STAGE_STYLE fallback ?? 미작동
?? 사용 시 null/undefined만 처리 — 빈 문자열('')은 통과. STAGE_STYLE[''] = undefined → .stageBg 접근 시 에러.
avatarIdx 제거 ✅ 
// 수정 전
AVATAR_POOL[c.avatarIdx % AVATAR_POOL.length]
// 수정 후
AVATAR_POOL[sorted.indexOf(c) % AVATAR_POOL.length]
AVATAR_POOL[couples.indexOf(selectedCouple) % AVATAR_POOL.length]
fallback ?? → || ✅ 
// 수정 전 — null/undefined만 처리
const st = STAGE_STYLE[c.stage] ?? STAGE_STYLE['초기상담'];
// 수정 후 — null, undefined, 빈 문자열 모두 처리
const st = STAGE_STYLE[c.stage] || STAGE_STYLE['초기상담'];
