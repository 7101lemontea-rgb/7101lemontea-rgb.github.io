01 — Supabase Migration Errors

Supabase 연동 에러

planners.json에서 Supabase wedding_planners 테이블로 마이그레이션하는 과정에서 실제로 발생한 에러들을 정리합니다. 각 에러의 발생 상황, 원인, 해결 방법, 코드 예시를 포함합니다.

#01   Cannot read property 'forEach' of null

Supabase에서 데이터를 받아오기 전, useMemo 내에서 배열 컬럼(activity_regions, specialties 등)에 접근할 때 발생합니다.

발생 파일 index.jsx
원인
Supabase 테이블 일부 row의 배열 컬럼이 null로 저장되어 있음. .forEach()는 null에 호출 불가.
해결
배열 접근 시 모두 ?. 옵셔널 체이닝을 사용하면 null일 때 undefined를 반환하고 에러를 방지함.

ERROR   Cannot read property 'forEach' of null

잘못된 코드 ❌ 
plannersData.forEach((planner) => {
  planner.activity_regions.forEach((r) => regions.add(r));
  planner.specialties.forEach((s) => specialties.add(s));
});
수정된 코드 ✅ 
plannersData.forEach((planner) => {
  planner.activity_regions?.forEach((r) => regions.add(r));    // ✅
  planner.specialties?.forEach((s) => specialties.add(s));      // ✅
  planner.style_keywords?.forEach((k) => styles.add(k));         // ✅
});

#02   useEffect must not return anything besides a function

useEffect 콜백에 async를 직접 붙여서 작성했을 때 발생합니다. async 함수는 항상 Promise를 반환하는데, useEffect는 cleanup 함수(또는 아무것도 반환하지 않음)만 허용합니다.

ERROR   useEffect must not return anything besides a function

발생 파일 [id].jsx / index.jsx
원인
useEffect(async () => { ... }) 형태로 작성하면 Promise 객체가 반환되어 React가 경고/에러를 발생시킴.
해결
useEffect 콜백 내부에 async 함수를 별도로 선언하고 호출만 하는 패턴을 사용.
잘못된 코드 ❌ 
// ❌ async를 직접 전달 → Promise 반환
useEffect(async () => {
  const { data } = await supabase.from('wedding_planners').select('*');
  setPlannersData(data);
}, []);
수정된 코드 ✅ 
// ✅ 내부에서 async 함수를 선언하고 호출만
useEffect(() => {
  async function fetchPlanners() {
    const { data, error } = await supabase
      .from('wedding_planners')
      .select('*');
    if (!error && data) setPlannersData(data);
    setLoading(false);
  }
  fetchPlanners();  // 호출만, return 안 함
}, []);

#03   필터/목록에 플래너 데이터가 표시되지 않음

Supabase fetch는 성공(data 로그 확인)했으나 화면에 플래너 카드가 하나도 표시되지 않고 '일치하는 플래너가 없습니다' 메시지가 출력되는 증상입니다.

발생 파일 index.jsx
원인
filteredPlanners useMemo의 의존성 배열에 plannersData가 누락되어, 데이터가 도착해도 useMemo가 재실행되지 않고 초기 빈 배열([])로 필터링한 결과를 계속 사용함.
해결
filteredPlanners와 filterOptions 두 useMemo 모두 의존성 배열에 plannersData를 추가.
잘못된 코드 ❌ 
const filteredPlanners = useMemo(() => {
  return plannersData.filter(...);
}, [searchText, selectedFilters]);  // ❌ plannersData 누락

const filterOptions = useMemo(() => {
  ...
}, []);  // ❌ plannersData 누락
수정된 코드 ✅ 
const filteredPlanners = useMemo(() => {
  return plannersData.filter(...);
}, [searchText, selectedFilters, plannersData]);  // ✅

const filterOptions = useMemo(() => {
  ...
}, [plannersData]);  // ✅

#04   Hook 관련 에러 (조건부 return 위치)

로딩 처리를 위해 if (loading) return ...을 추가했더니 이전에 없던 Hook 관련 에러가 발생하는 증상입니다.

발생 파일 [id].jsx / index.jsx
원인
if (loading) return을 useEffect 선언보다 앞에 배치하면, 조건에 따라 Hook이 건너뛰어지는 상황이 발생. React는 Hook이 매 렌더링에서 동일한 순서로 호출되어야 한다는 Rules of Hooks를 강제함.
해결
모든 useState, useEffect, useMemo 선언을 먼저 작성하고, 조건부 return은 반드시 마지막에 배치.
잘못된 코드 ❌ 
export default function Screen() {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  // ❌ Hook 앞에 return
  if (loading) return <ActivityIndicator />;
  useEffect(() => { ... }, []);  // ← 조건부 실행 → 에러!
}
수정된 코드 ✅ 
export default function Screen() {
  // 1. 모든 useState
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  // 2. 모든 useEffect
  useEffect(() => { ... }, []);
  // 3. 모든 useMemo
  const filtered = useMemo(() => { ... }, [data]);
  // 4. 조건부 return은 항상 마지막 ✅
  if (loading) return <ActivityIndicator />;
  if (!data) return <Text>데이터 없음</Text>;
  return <View>...</View>;
}
02 — Mypage & Auth Errors

마이페이지 & 인증 에러

Mongle 앱의 마이페이지(mypage.jsx) 개발 과정에서 발생한 6건의 트러블슈팅을 기록합니다. DB 테이블 구조는 아래와 같습니다. 

auth.users (Supabase 기본)
└── id ──────────────────────────────┐
                                     ↓
user_profiles
├── id         ← auth.users.id 와 1:1 매핑
├── role       ← "planner" | "couple"
├── planner_id ─→ wedding_planners.id

wedding_planners

couples

#05   마이 탭 눌러도 화면 이동 안 됨

ReferenceError: Property 'session' doesn't exist  — 발생 위치: TouchableOpacity.props.onPress (app/(couple)/_layout.jsx)

발생 파일 (couple)/_layout.jsx
원인
tabBarButton으로 onPress를 재정의하면 Expo Router의 기본 탭 전환 로직이 완전히 차단됨. 또한 session 변수를 선언하지 않고 onPress 내에서 참조.
해결
tabBarButton 완전 제거하여 Expo Router 기본 탭 전환에 위임. session은 useEffect + useState로 별도 관리. 로그인 체크는 mypage.jsx 내부에서 처리.
잘못된 코드 ❌ 
// ❌ tabBarButton이 Expo Router 탭 전환을 막음
<Tabs.Screen name="mypage" options={{
  tabBarButton: (props) => (
    <TouchableOpacity {...props} onPress={() => {
      if (session) {  // ← session 미선언 → ReferenceError
        router.push("/(couple)/mypage");
      } else {
        router.push("/(auth)/login");
      }
    }} />
  ),
}} />
수정된 코드 ✅ 
// ✅ tabBarButton 없이 단순하게
const [session, setSession] = useState(null);
useEffect(() => {
  supabase.auth.getSession().then(({ data: { session } }) => {
    setSession(session);
  });
  const { data: { subscription } } = supabase.auth.onAuthStateChange(
    (_event, session) => setSession(session)
  );
  return () => subscription.unsubscribe();
}, []);

// tabBarButton 완전 제거
<Tabs.Screen name="mypage" options={{
  title: "마이",
  tabBarIcon: ({ color, size }) => (
    <Ionicons name="person-outline" size={size} color={color} />
  ),
}} />

#06   로딩 스피너가 화면을 막아 마이페이지 진입 불가

마이 탭을 눌러도 로딩 스피너만 표시되거나, 빈 화면이 유지됩니다.

발생 파일 mypage.jsx
원인
loading 초기값을 true로 설정하여, 세션 확인이 완료되기 전부터 ActivityIndicator가 렌더링되어 화면 전체를 차단. 세션이 없을 때 router.replace로 강제 이동시켜 정상적인 탭 전환도 방해.
해결
loading 초기값을 false로 변경. 세션 없을 때 redirect 없이 그냥 return. 프로필 로드 시작할 때만 setLoading(true) 호출.
잘못된 코드 ❌ 
// ❌ loading 초기값 true → 세션 확인 전 스피너가 화면 차단
const [loading, setLoading] = useState(true);
useEffect(() => {
  supabase.auth.getSession().then(({ data: { session } }) => {
    if (!session) {
      router.replace("/(auth)/login");  // ← 탭 전환 방해
      return;
    }
    setSession(session);
    loadProfile(session.user.id);
  });
}, []);
수정된 코드 ✅ 
// ✅ loading 초기값 false, 세션 없을 때 redirect 제거
const [loading, setLoading] = useState(false);  // ← false로 변경
useEffect(() => {
  supabase.auth.getSession().then(({ data: { session } }) => {
    if (!session) return;  // ← redirect 없이 그냥 return
    setSession(session);
    setLoading(true);  // ← 프로필 로드 시작할 때만 true
    loadProfile(session.user.id);
  });
}, []);

#07   커플 계정 로그인 시 아무 반응 없음

커플 계정으로 로그인 버튼을 눌러도 오류 메시지도, 화면 전환도, alert도 아무것도 나타나지 않는 증상. 플래너 계정은 정상 로그인됩니다.

발생 파일 login.jsx
원인
user_metadata.role로 계정 유형을 구분하려 했으나, 가입 시 user_metadata에 role을 저장하지 않아 값이 undefined. 두 조건 모두 false가 되고, signOut 없이 return하여 로그인된 채로 아무것도 실행되지 않음.
해결
fetchUserRole로 DB에서 직접 role 조회. user_profiles 테이블을 단일 진실 공급원(Single Source of Truth)으로 사용. 역할 불일치 시 반드시 signOut() 호출.
잘못된 코드 ❌ 
// ❌ user_metadata.role이 undefined
const userRole = data.user?.user_metadata?.role;  // → undefined
// 두 조건 모두 false → 아무것도 실행 안 됨
if (userRole === "planner" && !isPlanner) { ... }  // false
if (userRole === "couple" && isPlanner) { ... }    // false
// signOut 없이 return → 로그인 상태로 멈춤
수정된 코드 ✅ 
// ✅ fetchUserRole로 DB에서 직접 role 조회
import { signIn, signOut, fetchUserRole } from '../../lib/auth';

const handleLogin = async () => {
  const data = await signIn(email, password);
  const profile = await fetchUserRole(data.user.id);
  const userRole = profile.role;  // "planner" | "couple"

  if (userRole === "planner" && !isPlanner) {
    await signOut();  // ← 로그인 상태 반드시 되돌리기
    showAlert("웨딩 플래너 계정입니다.\n웨딩 플래너 탭을 선택해주세요.");
    return;
  }
  if (userRole === "couple" && isPlanner) {
    await signOut();  // ← 마찬가지로 signOut 필수
    showAlert("예비 신혼 계정입니다.\n예비 신혼 탭을 선택해주세요.");
    return;
  }
  // 정상 → AuthGate onAuthStateChange가 자동 화면 전환
};

#08   플래너 계정이 마이페이지 접근 시 튕김

플래너 계정으로 로그인 후 마이 탭을 누르면 (couple)/mypage 대신 (couple) 홈으로 튕겨지는 증상입니다.

발생 파일 AuthGate
원인
AuthGate의 planner 분기에서 inCoupleGroup에 대한 예외가 없어, 플래너가 (couple)/mypage에 있을 때 inPlannerGroup이 false가 되어 /(couple)로 리다이렉트.
배경
mypage.jsx는 app/(couple)/ 디렉토리에 위치하지만, 플래너와 커플 모두 접근할 수 있도록 설계되어 있어 플래너가 (couple) 경로에 있는 것이 허용되어야 함.
잘못된 코드 ❌ 
// ❌ 플래너가 (couple) 경로에 있으면 무조건 튕겨냄
if (role === "planner") {
  if (!inPlannerGroup && !inVendorsGroup && !inPlannersGroup)
  // inCoupleGroup 예외 없음 → (couple)/mypage 접근 시 튕김
    router.replace("/(couple)");
}
수정된 코드 ✅ 
// ✅ inCoupleGroup 예외 추가
if (role === "planner") {
  if (!inPlannerGroup && !inCoupleGroup && !inVendorsGroup && !inPlannersGroup)
  // ↑ 추가 → 플래너가 (couple)/mypage 접근 허용
    router.replace("/(couple)");
}
03 — Core Principles

핵심 원칙 요약

1. React Hook 순서 규칙 (Rules of Hooks)

📌 필수 규칙   Hook은 항상 컴포넌트 최상단에서 동일한 순서로 호출되어야 합니다. 

export default function Screen() {
  // Step 1. 모든 useState 먼저
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  // Step 2. 모든 useEffect
  useEffect(() => { ... }, []);
  // Step 3. 모든 useMemo / useCallback
  const result = useMemo(() => { ... }, [data]);
  // Step 4. 조건부 return — 항상 마지막!
  if (loading) return <ActivityIndicator />;
  if (!data) return <Text>없음</Text>;
  return <View>...</View>;
}

2. useEffect 내 async 사용 패턴

⚠️ 주의   useEffect에 async를 직접 전달하면 Promise가 반환되어 에러가 발생합니다. 

// ❌ 잘못된 방식
useEffect(async () => {
  const data = await fetch(...);
}, []);

// ✅ 올바른 방식
useEffect(() => {
  async function fetchData() {
    const data = await fetch(...);
    setState(data);
  }
  fetchData();
}, []);

3. 옵셔널 체이닝 (?.) 적용 대상

Supabase 데이터는 일부 컬럼이 null일 수 있습니다. 배열 및 객체 접근 시 항상 ?.를 사용하세요. 

// 배열 컬럼
planner.activity_regions?.join(", ")
planner.specialties?.map(...)
planner.style_keywords?.forEach(...)
// 중첩 객체
planner.service_scope?.included?.map(...)  // 두 번 필요
// 숫자 메서드
planner.base_price_krw?.toLocaleString()

4. useMemo 의존성 관리

💡 팁   useMemo 내에서 사용하는 state는 모두 의존성 배열에 포함해야 값이 최신 상태로 유지됩니다. 

// ❌ plannersData 누락 → 데이터 도착 후에도 재계산 안 됨
const filtered = useMemo(() => {
  return plannersData.filter(...);
}, [searchText, selectedFilters]);

// ✅ 모든 참조 변수 포함
const filtered = useMemo(() => {
  return plannersData.filter(...);
}, [searchText, selectedFilters, plannersData]);

5. Supabase 테스트 데이터 필터링

wedding_planners 테이블에 brand_name이 null인 테스트 계정이 존재하면 렌더링 에러가 발생합니다. fetch 쿼리에 필터를 추가하세요. 

const { data, error } = await supabase
  .from('wedding_planners')
  .select('*')
  .not('brand_name', 'is', null)  // ✅ 불완전한 row 제외
  .order('created_at', { ascending: false });

6. role 구분 — user_metadata vs user_profiles

📌 핵심 원칙   user_metadata는 가입 시 별도로 저장해야 하므로 신뢰할 수 없습니다. role 구분은 user_profiles 테이블을 단일 진실 공급원(Single Source of Truth)으로 사용하세요.

04 — Checklist

재발 방지 체크리스트

재발 방지 체크리스트

Check 01 Supabase 테이블명 확인
Supabase 테이블명은 대시보드에서 반드시 확인 후 코드에 작성. 파일명(user_profiles_rows.csv 등)이 힌트가 될 수 있음.
Check 02 role은 user_profiles 테이블을 단일 소스로
role 구분은 user_metadata가 아닌 user_profiles 테이블을 단일 소스로 사용.
Check 03 tabBarButton 재정의 금지
Expo Router에서 tabBarButton으로 onPress를 재정의하면 탭 전환이 차단됨. 인증 체크는 화면 내부에서 처리할 것.
Check 04 로그인 실패 시 signOut() 필수
역할 불일치 등 로그인 실패 시 반드시 signOut()을 호출하여 세션 상태를 정리.
Check 05 AuthGate 수정 시 전체 조합 테스트
AuthGate 리다이렉트 조건 수정 시 모든 role × 경로 조합(planner/couple × auth/couple/planner/vendors)을 테스트.
