수정 전 ❌ 
const inSettingsGroup = rootSegment === 'settings';
...
} else if (!inCoupleGroup) {

수정 후 ✅

A2. 플래너 상세 페이지 Supabase 조회 오류

대상 파일: app/(couple)/planners/[id].jsx

문제 1 fetchPlanner 0 rows 반환 (PGRST116)
원인
.eq('name', id) 조회 시 라우트 파라미터 값 불일치. planners.json의 id 필드가 name 컬럼을 참조하고 있었음.
해결
mapPlanner의 id: p.name 참조 방식 확인 후 테이블에 name = 'planner1' 행 존재 → 조회 정상 확인.
문제 2 user_profiles Join 실패
원인
user_profiles!planner_id Join 방향 오류. 실제 외래키는 user_profiles.planner_id → wedding_planners.id인데 역방향 Join 시도로 결과 없음.
해결
Join 제거 후 .select('*')으로 단순 조회. 수신자 ID는 handleConsultSubmit 내부에서 별도 처리.

최종 fetchPlanner 코드입니다.

useEffect(() => {
  async function fetchPlanner() {
    const { data, error } = await supabase
      .from('wedding_planners')
      .select('*')
      .eq('name', id)
      .single();
    if (error) { console.error('조회 에러:', error); }
    else { setSelectedPlanner(data); }
    setLoading(false);
  }
  fetchPlanner();
}, [id]);

A3. 상담 신청 실패 — 사용자 정보 식별 불가

대상 파일: app/(couple)/planners/[id].jsx — handleConsultSubmit 함수  ·  증상: "사용자 정보를 식별할 수 없습니다" 에러

1단계 user_profiles Join 결과 빈 배열 반환
Join 쿼리 방식 문제 → A2에서 해결.
2단계 별도 쿼리 변경 후에도 0 rows 반환
원인
RLS(Row Level Security) 정책 미설정. couple 유저가 planner의 user_profiles 행을 SELECT할 권한 없음. SQL Editor 직접 실행 시 정상 결과 → RLS 차단 확인.
해결
Supabase SQL Editor에서 RLS 정책 추가. handleConsultSubmit의 수신자 ID 조회 방식을 별도 쿼리로 분리.

관련 DB 구조

테이블 컬럼 설명
wedding_planners id (UUID, PK)
name 식별자 (라우트 파라미터)
user_profiles id (UUID, PK) = auth.uid()
role 'couple' | 'planner'
planner_id (FK) → wedding_planners.id
couple_id (FK) → couples.id
RLS 정책 추가 SQL 
CREATE POLICY "Anyone can read planner profiles"
  ON user_profiles FOR SELECT
  USING (role = 'planner');
handleConsultSubmit 수정 ✅ 
// 수정 전
const plannerUserId = selectedPlanner?.user_profiles?.[0]?.id;

// 수정 후 — 별도 쿼리 분리
const { data: userProfile, error: profileError } = await supabase
  .from('user_profiles')
  .select('id')
  .eq('planner_id', selectedPlanner.id)
  .single();
if (profileError || !userProfile) {
  throw new Error('플래너 계정 정보를 찾을 수 없습니다.');
}
await sendNotificationToPlanner({
  plannerId: userProfile.id,
  plannerName: selectedPlanner.name, ...
});

B3. 화면별 연동 작업 내용

B3.1 dashboard.jsx — 플래너 대시보드

연동 테이블: wedding_planners, couples, couple_schedules, planner_vendors, couple_payments

  • 하드코딩 임시 데이터 전체 제거. useAuth()에서 planner_id를 꺼내 모든 쿼리 필터로 사용.
  • 5개 테이블 Realtime 구독 — 변경 시 자동 갱신.
  • 진행률(%): couples.progress가 0이면 couple_payments 합계로 동적 계산.
  • 수금/미수금: couple_payments 기반 집계.
  • 협력 업체: planner_vendors에서 유형별 그룹화, 최대 3개 유형 표시.
오늘 할 일 카드 동작 방식
  • 표시 순서: 오늘 미완료(urgent) → 이후 미완료(날짜순) → 최대 5개
  • useRef(lockedTodoIds)로 최초 로드 시 5개 ID 고정
  • 완료 토글 시 그 자리에서 취소선 표시 (항목 유지)
  • "완료 지우기" 버튼: 완료 항목 있을 때만 노출
  • 완료 지우기 후 lockedTodoIds 초기화 → fetchTodos() 재호출 → 새 5개 채움

B3.2 couple_list.jsx — 커플 목록

연동 테이블: couples, couple_schedules

  • INITIAL_COUPLES, INITIAL_SCHEDULES 하드코딩 제거.
  • DB 날짜(YYYY-MM-DD) ↔ 표시용(YYYY년 M월 D일) 변환 유틸 추가.
  • 커플 추가/수정: supabase.insert / .update. 커플 삭제: supabase.delete.
  • 일정 동기화: new- 접두사로 신규/기존 구분, 수정 완료 시 일괄 반영.
  • 필드명 변경: c.name → groom_name/bride_name, c.date → wedding_date.
  • avatarIdx 제거: sorted.indexOf(c) % AVATAR_POOL.length로 대체.

⚠️ stage 값 주의 — DB 저장값에 띄어쓰기 없음: '최종점검' / '준비중' / '초기상담' / '계약완료'. STAGE_STYLE fallback은 ?? 대신 || 사용 (빈 문자열도 처리).

B3.3 planner_todo_list.jsx — 할 일 목록

연동 테이블: couple_schedules, couples

  • ALL_TODOS, COUPLE_OPTIONS 하드코딩 제거.
  • 필드명: item.text → item.title, 날짜: item.dueDate(YYYYMMDD) → item.scheduled_date(YYYY-MM-DD).
  • urgent 판단: scheduled_date === todayDb.
  • 완료 토글: 낙관적 업데이트 + supabase.update, 실패 시 롤백.

B3.4 planner_budget.jsx — 비용 현황

연동 테이블: couples, couple_payments, couple_vendor_costs

  • 수금 항목 추가: couple_payments.insert (due_date 컬럼). 업체 지출 추가: couple_vendor_costs.insert.

B5. 공통 개발 패턴

planner_id 사용 방법

하드코딩 절대 금지 — 반드시 AuthContext에서 동적으로 가져올 것 

const { planner_id } = useAuth();
useEffect(() => {
  if (!planner_id) return;  // early return 필수
  fetchData();
}, [planner_id]);

낙관적 업데이트 패턴

// 1. UI 즉시 반영
setTodos(prev => prev.map(t => t.id === id ? {...t, done: !done} : t));
// 2. DB 업데이트
const { error } = await supabase.from(...).update({done: !done}).eq('id', id);
// 3. 실패 시 롤백
if (error) setTodos(prev => prev.map(t => t.id === id ? {...t, done} : t));

날짜 변환 규칙

형식 설명 / 용도
YYYY-MM-DD DB 저장값 (예: 2026-04-05)
YYYY년 M월 D일 화면 표시 (예: 2026년 4월 5일)
YYYYMMDD 폼 입력값 (planner_todo_list)

avatarIdx 대체

// Before: DB에 없는 컬럼
AVATAR_POOL[c.avatarIdx % AVATAR_POOL.length]
// After: 배열 인덱스로 대체
AVATAR_POOL[sorted.indexOf(c) % AVATAR_POOL.length]

C7. 잔여 작업 및 핵심 주의사항

잔여 작업

  • Row Level Security(RLS) 정책 전체 설정

  • couples.stage 값 통일 (DB에 '준비중' 형태로 저장되도록 앱 전체 확인)

  • NotificationProvider가 (couple) 레이아웃을 감싸고 있는지 확인

RLS 정책 권장 SQL

-- 커플: 자신의 일정만 조회
CREATE POLICY "couple read own schedules" ON couple_schedules
  FOR SELECT USING (couple_id = auth.uid());

-- 플래너: 담당 커플 전체 관리
CREATE POLICY "planner manage schedules" ON couple_schedules
  FOR ALL USING (planner_id = auth.uid());
