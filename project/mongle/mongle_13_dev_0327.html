Overview

프로젝트 개요

Mongle은 웨딩 플래너와 예비 신혼부부를 연결하는 웨딩 통합 관리 플랫폼입니다. 오늘 작업은 크게 세 축으로 진행되었습니다. Supabase DB 11개 테이블 신규 설계 및 Auth 트리거 구현, 기존 로컬 JSON 파일에서 Supabase 실데이터로의 마이그레이션, 그리고 마이페이지 개발과 인증 흐름 수정입니다.

서비스 목적

웨딩 플래너와 예비 신혼부부를 연결하는 웨딩 통합 관리 플랫폼

개발 환경

React Native (Expo Router)

데이터베이스

Supabase (PostgreSQL)

인증

Supabase Auth (이메일 + 비밀번호)

사용자 역할

웨딩 플래너 (planner) · 예비 신혼부부 (couple)

가입 시나리오

시나리오 설명 planner_id user_id
A 플래너가 먼저 커플 등록 O NULL
B 커플이 직접 가입 NULL O
C 플래너 매칭 후 커플 가입 O O
01 — Supabase DB Design

Supabase DB 설계

기존 wedding_planners 테이블은 그대로 유지하고, 아래 11개 테이블을 신규 추가했습니다. 구현 스크립트는 supabase_add_new_tables_only.sql에 정리되어 있습니다.

1-1. 테이블 목록

테이블명 역할 및 설명
wedding_planners 플래너 프로필 정보 — 더미 데이터 포함, Auth와 별도 관리 (기존 유지)
user_profiles auth.users와 1:1 연결 — 플래너/커플 역할 관리
couples 커플 기본 정보 (planner_id, user_id 모두 nullable)
couple_payments 수금 항목 (계약금/중도금/잔금 등)
couple_schedules 커플 일정 관리
planner_vendors 플래너가 관리하는 협력 업체 (기존 vendors와 분리)
vendor_costs 업체별 지출 (planner_vendors ↔ couples 연결)
todos 플래너 할 일 목록
weddings 타임라인·비용 서비스 루트 (couples와 1:1 연결)
timeline_items 타임라인 일정 (플래너·커플 공동 편집)
cost_items 비용 항목 (만원 단위)
notification_types 알림 종류 마스터 데이터
notifications 플래너·커플 양쪽 알림 통합 관리

1-2. 테이블 관계 구조

wedding_planners ←── couples.planner_id         (nullable)
auth.users       ←── couples.user_id            (nullable)

couples          ──→ weddings.couple_id         (1:1 UNIQUE)
couples          ←── couple_payments.couple_id
couples          ←── couple_schedules.couple_id
couples          ←── vendor_costs.couple_id

wedding_planners ←── planner_vendors.planner_id
planner_vendors  ←── vendor_costs.vendor_id
wedding_planners ←── todos.planner_id

weddings         ←── timeline_items.wedding_id
weddings         ←── cost_items.wedding_id
weddings         ←── notifications.wedding_id

1-3. ENUM 타입 및 공동 편집 권한

ENUM 값
timeline_status done / active / next / future
item_color rose / sage / lavender / sky / peach / mint

timeline_items.editable_by 컬럼은 공동 편집 권한을 제어합니다.

값 설명
both (기본값) 플래너·커플 모두 수정 가능
planner_only 플래너만 수정 가능 (계약·공식 일정)
couple_only 커플만 수정 가능 (개인 메모성 일정)

1-4. 알림 종류 (notification_types)

ID 레이블 수신 대상
todo_deadline 할 일 마감 플래너
payment_due 잔금 납부 임박 플래너·커플
dday_reminder D-Day 알림 플래너·커플
todo_completed 할 일 완료 플래너
couple_added 새 커플 등록 플래너
timeline_added 일정 추가 플래너·커플
timeline_updated 일정 변경 플래너·커플
cost_urgent 긴급 잔금 마감 커플
02 — Supabase Auth Design

Supabase Auth 설계

2-1. 가입 흐름

플래너 가입 planner

signUp({ data: { role: 'planner' } }) → 트리거 → wedding_planners 행 생성 + user_profiles 생성

커플 가입 couple

signUp({ data: { role: 'couple' } }) → 트리거 → couples 행 생성 + user_profiles 생성

가입 시 이메일만 저장되며, 나머지 정보(이름, 웨딩일 등)는 이후 앱에서 입력합니다.

2-2. Auth 트리거 함수

함수명 역할
handle_new_user() 통합 트리거 — raw_user_meta_data.role 값으로 플래너/커플 분기
handle_new_planner() wedding_planners 신규 행 생성 + user_profiles 생성 (role='planner')
handle_new_couple() couples 신규 행 생성 + user_profiles 생성 (role='couple')

2-3. 로그인 후 역할 분기

로그인 성공
  → user_profiles에서 role 조회
  → role = 'planner'  →  /(planner) 화면
  → role = 'couple'   →  /(couple)  화면
  → 세션 없음         →  /(auth)/login 화면

⚠️ 트러블슈팅 — 플래너 회원가입 오류: database error saving new user

원인 wedding_planners.raw_data NOT NULL 제약
문제
wedding_planners의 raw_data 컬럼이 NOT NULL 제약을 가지고 있으나, 트리거 함수 handle_new_planner()에서 해당 컬럼에 값을 INSERT하지 않아 오류 발생.
해결
NOT NULL 제약 조건을 제거하여 트리거가 최소 정보(이메일)만으로 행을 생성할 수 있도록 수정. name도 nullable로 변경 후 앱에서 입력받는 구조로 개선. 
-- raw_data NOT NULL 제약 제거
ALTER TABLE wedding_planners
  ALTER COLUMN raw_data DROP NOT NULL;

-- name도 nullable로 변경 (가입 후 앱에서 입력)
ALTER TABLE wedding_planners
  ALTER COLUMN name DROP NOT NULL;
03 — Supabase Migration

Supabase 마이그레이션

기존에 로컬 planners.json 파일에서 플래너 데이터를 가져오던 방식을 Supabase wedding_planners 테이블에서 실시간으로 fetch하는 방식으로 마이그레이션했습니다.

파일 변경 내용
app/planners/index.jsx 플래너 목록 화면 — 전체 데이터 fetch, 필터/검색
app/planners/[id].jsx 플래너 상세 화면 — name 기준 단건 fetch

3-1. 환경 설정

기존에 작성된 supabase.js 파일을 그대로 사용합니다. AsyncStorage 기반 세션 유지가 이미 설정되어 있어 별도 수정이 불필요합니다. 

// lib/supabase.js
import { createClient } from '@supabase/supabase-js'
import AsyncStorage from '@react-native-async-storage/async-storage'

const supabaseUrl = process.env.EXPO_PUBLIC_SUPABASE_URL
const supabaseAnonKey = process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY

export const supabase = createClient(supabaseUrl, supabaseAnonKey, {
  auth: {
    storage: AsyncStorage,
    autoRefreshToken: true,
    persistSession: true,
    detectSessionInUrl: false,
  },
})

⚠️ 주의   Next.js는 NEXT_PUBLIC_ 접두사를 사용하지만, Expo는 EXPO_PUBLIC_ 접두사를 사용합니다. 

# .env.local
EXPO_PUBLIC_SUPABASE_URL=https://xxxx.supabase.co
EXPO_PUBLIC_SUPABASE_ANON_KEY=eyJ...

3-2. index.jsx 마이그레이션

State 및 useEffect

3-3. [id].jsx 마이그레이션

변경 사항 요약: planners.json import 제거 → selectedPlanner를 useState로 관리 → useEffect로 name 기준 단건 fetch → 기존 BackHandler useEffect와 분리 → 로딩 처리 추가 → JSX 내 ?. 적용. 

export default function PlannerDetailScreen() {
  const { id } = useLocalSearchParams();  // id = planner.name 값

  const [selectedPlanner, setSelectedPlanner] = useState(null);
  const [loading, setLoading] = useState(true);
  // 기존 state들 그대로 유지
  const [modalVisible, setModalVisible] = useState(false);

  // ✅ useEffect 1 — Supabase fetch
  useEffect(() => {
    async function fetchPlanner() {
      const { data, error } = await supabase
        .from('wedding_planners')
        .select('*')
        .eq('name', id)
        .not('brand_name', 'is', null)
        .single();
      if (!error && data) setSelectedPlanner(data);
      setLoading(false);
    }
    fetchPlanner();
  }, [id]);

  // ✅ useEffect 2 — BackHandler (기존 코드 그대로, 완전히 분리)
  useEffect(() => {
    const handleBackPress = () => { ... };
    const subscription = BackHandler.addEventListener('hardwareBackPress', handleBackPress);
    return () => subscription.remove();
  }, []);

  // ✅ 모든 Hook 이후 조건부 return
  if (loading) return <View style={styles.errorContainer}><ActivityIndicator color="#c9a98e" /></View>;
}

04 — Mypage Development

마이페이지 개발

app/(couple)/mypage.jsx 파일로, 플래너와 커플 모두 접근 가능한 마이페이지입니다.

4-1. DB 테이블 구조

테이블명 주요 컬럼 역할
user_profiles id, role, planner_id, couple_id auth.users.id와 1:1 매핑. role과 각 상세 테이블 id 보관
wedding_planners id, name, brand_name, one_liner, specialties, style_keywords, career_years, weddings_completed 플래너 상세 정보
couples id, user_id, groom_name, bride_name, email, phone 커플 상세 정보 
user_profiles (.eq("id", userId))
  ├── role === "planner"  →  wedding_planners (.eq("id", planner_id))
  └── role === "couple"   →  couples          (.eq("id", couple_id))

4-2. role별 표시 항목

플래너 planner
  • 프로필 카드 (이름, 브랜드명, 이메일, role 뱃지)
  • 한 줄 소개 (one_liner)
  • 경력·성사 웨딩 통계 카드
  • 전문 분야 칩 (specialties)
  • 스타일 키워드 칩 (style_keywords)
  • 주요 경험 목록 (major_experiences)
  • 설정: 프로필 편집, 알림, 비밀번호 변경, 개인정보·이용약관
커플 couple
  • 프로필 카드 (이름, 이메일, role 뱃지)
  • 계정 정보: 신랑/신부 이름, 이메일
  • 설정: 프로필 설정, 알림, 비밀번호 변경, 개인정보·이용약관
  • 공통 — 로그아웃 버튼, 회원 탈퇴 버튼

4-3. 프로필 2단계 조회 코드

const loadProfile = async (userId) => {
  // 1단계: user_profiles에서 role 및 연결 ID 조회
  const { data: profileRow } = await supabase
    .from("user_profiles")
    .select("role, planner_id, couple_id")
    .eq("id", userId)
    .single();
  const { role, planner_id, couple_id } = profileRow;
  setRole(role);

  // 2단계: role에 따라 상세 테이블 조회
  if (role === "planner" && planner_id) {
    const { data } = await supabase
      .from("wedding_planners")
      .select("name, brand_name, one_liner, ...")
      .eq("id", planner_id).single();
    setProfile(data);
  } else if (role === "couple" && couple_id) {
    const { data } = await supabase
      .from("couples")
      .select("groom_name, bride_name, email, phone")
      .eq("id", couple_id).single();
    setProfile(data);
  }
};

4-4. 핵심 수정 사항

loading 초기값

true → false로 변경. 초기값이 true이면 세션 확인 전에 로딩 스피너가 화면을 막아 탭 전환이 안 됨.

router.replace 제거

세션 없을 때 router.replace("/(auth)/login") 제거. _layout의 tabBarButton 제거 후 Expo Router가 자동 처리.

테이블명 수정

초기 "profiles" → "user_profiles"로 수정 (실제 DB 테이블명 반영). handleDeleteAccount 내부도 동일하게 수정.

05 — Layout & Login

5-1. tabBarButton 제거 — _layout.jsx

기존 코드에서 mypage 탭에 tabBarButton을 커스텀하여 onPress를 재정의했는데, 이 방식이 Expo Router의 기본 탭 네비게이션을 완전히 가로막아 마이페이지 화면이 열리지 않았습니다.

기존 코드 ❌ 
// ❌ tabBarButton이 Expo Router 탭 전환을 막음
<Tabs.Screen name="mypage" options={{
  tabBarButton: (props) => (
    <TouchableOpacity {...props} onPress={() => {
      if (session) {  // ← session 미선언 → ReferenceError
        router.push("/(couple)/mypage");
      } else {
        router.push("/(auth)/login");
      }
    }} />
  ),
}} />
수정된 코드 ✅ 
// ✅ tabBarButton 제거, session은 useEffect로 관리
const [session, setSession] = useState(null);
useEffect(() => {
  supabase.auth.getSession().then(({ data: { session } }) => {
    setSession(session);
  });
  const { data: { subscription } } = supabase.auth.onAuthStateChange(
    (_event, session) => setSession(session)
  );
  return () => subscription.unsubscribe();
}, []);

// mypage 탭 — tabBarButton 없이 단순하게
<Tabs.Screen name="mypage" options={{
  title: "마이",
  tabBarIcon: ({ color, size }) => (
    <Ionicons name="person-outline" size={size} color={color} />
  ),
}} />

5-2. fetchUserRole 적용 — login.jsx

기존 handleLogin에서 user_metadata.role로 계정 유형을 구분하려 했으나, 가입 시 user_metadata에 role을 저장하지 않아 값이 undefined였습니다. 그 결과 커플 계정으로 로그인해도 두 조건 모두 false가 되어 아무 반응이 없었습니다.

📌 핵심 원칙   user_metadata는 가입 시 별도로 저장해야 하므로 신뢰할 수 없습니다. role 구분은 user_profiles 테이블을 단일 진실 공급원(Single Source of Truth)으로 사용하세요.

기존 코드 ❌ 
// ❌ user_metadata.role이 undefined → 조건 모두 false
const userRole = data.user?.user_metadata?.role;  // → undefined
if (userRole === "planner" && !isPlanner) { ... }  // false
if (userRole === "couple" && isPlanner)  { ... }  // false
// signOut 없이 return → 로그인된 채로 아무것도 안 일어남
수정된 코드 ✅ 
// ✅ fetchUserRole로 DB에서 직접 role 조회
import { signIn, signOut, fetchUserRole } from '../../lib/auth';

const handleLogin = async () => {
  const data = await signIn(email, password);
  const profile = await fetchUserRole(data.user.id);
  const userRole = profile.role;  // "planner" | "couple"

  if (userRole === "planner" && !isPlanner) {
    await signOut();  // ← 로그인 상태 반드시 되돌리기
    showAlert("웨딩 플래너 계정입니다.\n웨딩 플래너 탭을 선택해주세요.");
    return;
  }
  if (userRole === "couple" && isPlanner) {
    await signOut();
    showAlert("예비 신혼 계정입니다.\n예비 신혼 탭을 선택해주세요.");
    return;
  }
  // 정상 → AuthGate의 onAuthStateChange가 자동 화면 전환
};
06 — AuthGate

AuthGate 리다이렉트 수정

AuthGate의 리다이렉트 로직에서 두 가지 문제가 발견되었습니다. 커플 로그인 후 홈 화면으로 자동 이동하지 않는 문제와, 플래너가 (couple)/mypage에 접근하면 홈으로 튕기는 문제를 모두 수정했습니다.

문제 1 커플 로그인 후 홈 화면 자동 이동 안 됨
원인
couple 분기 리다이렉트 조건에 inAuthGroup이 예외로 포함되어, 로그인 후 (auth) 화면에 있을 때 /(couple)로 이동하는 조건이 false가 됨.
해결
couple 분기에서 inAuthGroup 예외 제거.
문제 2 플래너가 (couple)/mypage 접근 시 튕김
원인
planner 분기에 inCoupleGroup 예외가 없어, 플래너가 (couple)/mypage에 있을 때 inPlannerGroup이 false가 되어 /(couple)로 리다이렉트됨. mypage.jsx는 플래너·커플 모두 접근 가능하도록 설계됨.
해결
planner 분기에 inCoupleGroup 예외 추가.
기존 코드 ❌ 
if (role === "planner") {
  if (!inPlannerGroup && !inVendorsGroup && !inPlannersGroup)
  // inCoupleGroup 예외 없음 → (couple)/mypage 접근 시 튕김
    router.replace("/(couple)");
} else {  // couple
  if (!inCoupleGroup && !inAuthGroup && !inVendorsGroup && !inPlannersGroup)
  // inAuthGroup 예외 → 로그인 후 (auth)에서 이동 안 됨
    router.replace("/(couple)");
}
수정된 코드 ✅ 
if (role === "planner") {
  if (!inPlannerGroup && !inCoupleGroup && !inVendorsGroup && !inPlannersGroup)
  // ↑ 추가 → 플래너가 (couple)/mypage 접근 허용
    router.replace("/(couple)");
} else {  // couple
  if (!inCoupleGroup && !inVendorsGroup && !inPlannersGroup)
  // inAuthGroup 제거 → 로그인 후 (auth)에서 (couple)로 자동 이동
    router.replace("/(couple)");
}

⚠️ 주의   inAuthGroup 예외를 제거하면 비로그인 상태에서 (auth) 화면이 정상 유지되는지 확인 필요. session이 없는 경우 분기가 먼저 실행되어 (auth) 접근은 여전히 허용됨.

07 — Query Reference

주요 쿼리 참고

SELECT role, planner_id, couple_id

플래너 담당 커플 목록

SELECT c.*, w.id AS wedding_id, w.budget
FROM couples c
LEFT JOIN weddings w ON w.couple_id = c.id
WHERE c.planner_id = (
  SELECT planner_id FROM user_profiles WHERE id = auth.uid()
) ORDER BY c.wedding_date;

커플 타임라인 조회

SELECT ti.* FROM timeline_items ti
JOIN weddings w ON w.id = ti.wedding_id
WHERE w.couple_id = (
  SELECT couple_id FROM user_profiles WHERE id = auth.uid()
) ORDER BY ti.event_date;

읽지 않은 알림 수

SELECT COUNT(*) FROM notifications
WHERE user_id = auth.uid() AND is_read = false;
