#01   미로그인 상태에서 무한 로딩

발생 파일 index.jsx
증상
로그인하지 않은 상태로 채팅 탭 진입 시 스피너가 멈추지 않고 화면이 계속 로딩 중으로 표시됨.
원인
supabase.auth.getUser()가 null을 반환해도 loading 상태를 false로 전환하는 코드가 없어 loading이 true인 채로 멈춤.
해결
useFocusEffect 내에서 user가 null이면 즉시 setIsAuthenticated(false)와 setLoading(false)를 호출하고 데이터 fetch를 건너뜀. 인증 상태를 null(확인 중) / true / false 세 단계로 분리.
수정 전 ❌ 
const load = async () => {
  const pid = await fetchProjectId();
  await fetchRooms(pid);
};
수정 후 ✅ 
const load = async () => {
  const { data: { user } } = await supabase.auth.getUser();
  if (!user) {
    setIsAuthenticated(false);
    setLoading(false);  // 반드시 호출
    return;
  }
  setIsAuthenticated(true);
  // ...이후 fetch 로직
};

💡 인증이 필요한 화면은 getUser() 결과를 먼저 확인하고, null 케이스에서 반드시 setLoading(false)를 호출해야 합니다.

#02   프로젝트 조회 시 .single() 에러

ERROR: Cannot coerce the result to a single JSON object

발생 파일 index.jsx
원인
Supabase .single()은 결과가 정확히 1건일 때만 성공하며, 0건이면 에러를 throw함. 신규 유저는 projects 테이블에 데이터가 없어 항상 0건이 반환됨.
해결
.single()을 .maybeSingle()로 변경. 결과가 0건일 때 에러 대신 null을 반환하여 신규 유저도 정상 처리됨.
수정 전 ❌ 
.limit(1).single();
// 0건이면 에러 throw
수정 후 ✅ 
.limit(1).maybeSingle();
// 0건이면 null 반환 (에러 없음)

#03   fetchRooms(null) 호출 후 무한 로딩

발생 파일 index.jsx
증상
프로젝트가 없는 유저로 로그인 시 채팅 목록 화면이 계속 로딩 중으로 표시됨.
원인
fetchProjectId()가 null을 반환하면 fetchRooms(null)이 호출됨. pid가 null이면 return만 하고 setLoading(false)를 호출하지 않아 loading이 true인 채로 유지됨.
해결
fetchRooms에서 pid가 null일 경우 setLoading(false)를 먼저 호출한 후 return 처리.
수정 전 ❌ 
const fetchRooms = async (pid) => {
  if (!pid) return;  // setLoading(false) 누락!
  // ...
};
수정 후 ✅ 
const fetchRooms = async (pid) => {
  if (!pid) {
    setLoading(false);  // 반드시 호출
    return;
  }
  // ...
};

#07   초대 코드 입력 시 '유효하지 않은 초대 코드' 오류

발생 위치 Supabase RLS
증상
올바른 초대 코드를 입력해도 '유효하지 않은 초대 코드입니다' Alert가 표시되며 채팅방 참여 실패.
원인
chats 테이블의 RLS 정책이 기존 멤버이거나 프로젝트 소유자인 경우만 SELECT를 허용함. 아직 멤버가 아닌 상태에서 invite_code로 조회하면 RLS에 막혀 0건이 반환됨.
해결
Supabase SQL Editor에서 invite_code IS NOT NULL 조건의 SELECT 정책을 추가하여 초대 코드가 있는 채팅방은 누구나 조회 가능하도록 변경. 
-- chats: 초대 코드로 채팅방 조회 정책 추가
CREATE POLICY "초대 코드로 채팅방 조회"
  ON chats FOR SELECT
  USING (invite_code IS NOT NULL);

#08   이메일 초대 시 '해당 이메일이 없다' 오류

발생 파일 [id].jsx
증상
멤버 초대 화면에서 이메일을 입력하고 초대 보내기를 누르면 '해당 이메일로 가입된 사용자를 찾을 수 없습니다' 오류 발생.
원인
profiles 테이블 구조 확인 결과 id, role, planner_id, couple_id, expo_push_token 컬럼만 존재하며 email 컬럼이 없음. 이메일로 유저를 조회하는 것이 구조적으로 불가능.
해결
이메일 초대 방식을 완전히 제거하고 초대 코드 단일 방식으로 전환. handleInviteByEmail, inviteTab, inviteEmail, inviting state, 이메일 탭 UI를 모두 제거.

💡 이메일 초대 기능을 다시 추가하려면 profiles 테이블에 email 컬럼 추가 및 auth.users 트리거 설정이 선행되어야 합니다.

#10   초대 모달 탭 전환 관련 렌더링 오류

발생 파일 [id].jsx
증상
이메일/코드 탭 전환 UI가 포함된 초대 모달에서 탭 전환 시 렌더링 불안정. #05 AIAssistantEntry 오류와 동일한 패턴.
원인
이메일 탭과 코드 탭을 조건부 렌더링으로 처리하던 구조가 state 변화에 따라 컴포넌트 트리가 변경되어 React 렌더링 최적화와 충돌 가능성 존재.
해결
이메일 탭 제거와 동시에 탭 전환 UI 전체를 삭제하고 단일 초대 코드 UI로 단순화. 탭 관련 state(inviteTab)도 함께 제거.
수정 전 ❌ 
// 탭 전환 구조
<View style={styles.tabRow}>
  <TouchableOpacity onPress={() => setInviteTab('email')}>이메일로 초대</TouchableOpacity>
  <TouchableOpacity onPress={() => setInviteTab('code')}>초대 코드</TouchableOpacity>
</View>
{inviteTab === 'email' && <EmailTab />}
{inviteTab === 'code'  && <CodeTab />}
수정 후 ✅ 
// 단일 초대 코드 UI
<View style={styles.codeSection}>
  // 초대 코드 생성 / 공유 UI만 표시
</View>

#7   새 채팅방 생성 후 무한 로딩

발생 파일 index.jsx
증상
채팅방 생성 모달에서 "만들기" 버튼을 눌렀을 때 로딩 스피너가 멈추지 않고, 모달이 닫히지 않음.
원인
handleCreateRoom 함수 내에서 각 에러 분기마다 setCreating(false)를 수동으로 호출하는 구조였으나, chat_members INSERT 이후 예외가 발생하는 경로에서 setCreating(false)가 누락되어 버튼이 영구 비활성화 상태 유지.
해결
handleCreateRoom 전체를 try-catch-finally로 감싸 성공·실패 모든 경우에 finally에서 setCreating(false) 보장.
수정 전 ❌ 
// 분기마다 수동 호출 — 누락 위험
setCreating(true);
if (error1) { setCreating(false); return; }
if (error2) { setCreating(false); return; }
// 중간 예외 발생 시 setCreating(false) 미호출
setCreating(false);
수정 후 ✅ 
// try-catch-finally 패턴
setCreating(true);
try {
  // ... 모든 로직
} catch (e) {
  Alert.alert('오류', '채팅방 생성 중 문제가 발생했습니다.');
} finally {
  setCreating(false);  // ✅ 반드시 실행됨
}

#8   Supabase 조인 — profiles 관계 없음 오류

Could not find a relationship between 'chat_members' and 'profiles' in the schema cache

발생 파일 [id].jsx
원인
Supabase의 테이블 조인 문법(tableName(columns))은 두 테이블 간에 Foreign Key가 정의되어 있어야 동작함. chat_members.user_id → profiles.id 간 외래키가 DB에 없어 스키마 캐시에서 관계를 찾지 못함.
방법 A (즉시 적용)
분리 조회 후 JS 병합 — 추가 DB 설정 불필요. (#11과 동일한 해결 방법)
방법 B (근본 해결)
Supabase에서 외래키 추가 후 조인 문법 사용 가능. 
-- 방법 B — 외래키 추가 SQL
ALTER TABLE chat_members
  ADD CONSTRAINT chat_members_user_id_fkey
  FOREIGN KEY (user_id) REFERENCES profiles(id) ON DELETE CASCADE;

-- 적용 후 아래 조인 문법 사용 가능
-- .select('user_id, role, profiles(display_name, email)')

💡 이번 작업에서는 방법 A로 해결. 외래키 없이도 동작하므로 추가 DB 마이그레이션 없이 즉시 배포 가능.

#9   RLS 정책 무한 재귀 오류

infinite recursion detected in policy for relation "chat_members"

발생 위치 Supabase RLS
원인
chat_members 테이블의 SELECT RLS 정책 내부에서 다시 chat_members를 SELECT하는 구조. "해당 chat_id에 내가 멤버인지" 확인하기 위해 같은 테이블을 서브쿼리로 참조하면 무한 재귀 발생.
해결
RLS 정책을 auth.uid() = user_id 단순 비교 방식으로 교체. 자기 자신을 참조하지 않으므로 재귀 없음.
기존 정책 ❌ 
-- 무한 재귀 발생
CREATE POLICY "members can view" ON chat_members
  FOR SELECT USING (
    auth.uid() IN (
      SELECT user_id FROM chat_members cm  -- ← 자기 자신 참조!
      WHERE cm.chat_id = chat_members.chat_id
    )
  );
수정된 정책 ✅ 
-- 기존 정책 모두 삭제
DROP POLICY IF EXISTS "members can view"    ON chat_members;
DROP POLICY IF EXISTS "owner can insert"   ON chat_members;
DROP POLICY IF EXISTS "members can insert" ON chat_members;
DROP POLICY IF EXISTS "owner can delete"   ON chat_members;

-- 단순 비교 — 재귀 없음
CREATE POLICY "members can view" ON chat_members
  FOR SELECT USING (auth.uid() = user_id);

CREATE POLICY "members can insert" ON chat_members
  FOR INSERT WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "members can delete" ON chat_members
  FOR DELETE USING (auth.uid() = user_id);

💡 Supabase RLS에서 동일 테이블을 참조하는 정책은 무한 재귀를 유발합니다. 같은 테이블의 다른 행을 확인해야 할 경우 SECURITY DEFINER 함수를 사용하거나 정책 설계를 단순화하세요.

04 — Prevention Checklist

재발 방지 체크리스트

Supabase

  • 단건 조회 시 결과가 0건일 수 있는 경우 .single() 대신 .maybeSingle() 사용
  • RLS 정책이 의도한 대로 동작하는지 SQL Editor에서 정책 목록을 먼저 확인
  • 미인증 상태에서도 접근이 필요한 조회(초대 코드 등)는 별도 SELECT 정책 추가
  • insert 시 state 변수 대신 최신 값을 담은 로컬 변수를 사용하여 타이밍 이슈 방지
  • Supabase 조인 전 두 테이블 간 Foreign Key 설정 여부 확인
  • RLS 정책 작성 시 동일 테이블 서브쿼리 사용 금지 — 무한 재귀 유발
  • 새 테이블 추가 시 RLS 활성화(ENABLE ROW LEVEL SECURITY) 확인
  • Supabase 쿼리 에러는 반드시 error 변수로 받아 console.error 출력

React Native

  • 비동기 데이터 로드 함수는 모든 분기(null, 에러, 성공)에서 setLoading(false)를 반드시 호출
  • async 함수에서 로딩 상태를 다룰 때는 try-catch-finally 패턴 사용
  • 인증이 필요한 화면은 getUser() 결과를 먼저 확인하고 null 케이스를 명시적으로 처리
  • 컴포넌트는 렌더 함수 내부에 선언하지 말 것 — 파일 최상단 또는 별도 파일로 분리
  • 버튼이 보이지만 동작하지 않으면 onPress 핸들러 연결 여부를 먼저 확인
  • chat_members 테이블 생성 후 기존 채팅방 데이터 마이그레이션 필수

Expo Router

  • router.push 경로는 실제 app/ 디렉터리 구조와 반드시 일치해야 함
  • 경로 오류 디버깅 시 에러 스택의 파일 경로를 확인하여 실제 위치 파악
