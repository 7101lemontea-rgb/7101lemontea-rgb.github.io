02 — Work Done

실제 수행한 작업

아이디어 브레인스토밍

그룹웨어 주제 후보 3개를 도출했습니다. 각 아이디어는 B2B·소비자 양방향 구조가 없다는 공통 시장 공백을 가지고 있었으며, 솔루션 분석을 통해 신규 서비스 기회를 확인했습니다.

💍
Idea 01
웨딩플래너 그룹웨어

플래너↔커플이 같은 화면에서 비용·일정·채팅을 함께 관리하는 서비스 부재

📚
Idea 02
자격증 준비 그룹웨어

문제풀이·학습 시간·커뮤니티가 각각 다른 앱에 분산, 통합 서비스 부재

🌸
Idea 03
꽃 예약 그룹웨어

소비자 주문 중계에만 집중, 꽃집 내부 업무관리(재고·주문·배달·정산) 도구 전무

솔루션 분석 및 주제 선정

3개 아이디어에 대해 기존 솔루션 총 7개(아이웨딩·다이렉트결혼준비·푸딩·웨드소프트, 맞추다·열품타, 꽃집청년들·플라워119)를 강점·한계 프레임워크로 분석했습니다. 분석 결과를 바탕으로 웨딩플래너 그룹웨어를 최종 프로젝트 주제로 선정했습니다.

선정 근거 ①
기존 4개 솔루션 모두 플래너(B2B)와 커플(소비자)이 같은 화면을 공유하는 기능이 전무했습니다. 시장 공백이 가장 명확했습니다.
선정 근거 ②
비용 투명화 + 일정 공유 + 실시간 협업 채팅이라는 세 가지 핵심 기능이 구체적으로 도출되어 MVP 범위를 명확히 정의할 수 있었습니다.
선정 근거 ③
웨드소프트(B2B)가 플래너 측 기능은 완성도 있게 구현했으나, 커플 측 접근을 차단하고 있어 양방향 구조로 차별화가 가능했습니다.
프로젝트명
Mongle — 웨딩플래너↔예비부부 공유 워크스페이스

화면 기능 계획서 작성

앱 화면을 크게 4개 영역으로 구분하고, 각 화면에서 필요한 핵심 기능을 정의했습니다.

메인 앱 화면 커플 공통
  • D-day 배너 — 결혼일까지 남은 일수 표시
  • 준비 진행률 요약 카드 (체크리스트 완료율)
  • 최근 채팅 메시지 미리보기
  • 예산 현황 요약 (확정 비용 / 총 예산 비율)
로그인 화면 인증
  • 소셜 로그인 (Google · Kakao · Apple)
  • 역할 선택 — 커플 / 플래너
  • 초대 코드 입력 (커플↔플래너 연결)
  • 온보딩 (최초 1회) — 서비스 핵심 가치 안내
일정 대시보드 커플 전용
  • 준비 타임라인 — 단계별 체크리스트 (드레스·스튜디오·홀 등)
  • 비용 명세서 — 플래너 입력 비용 실시간 확인, AI 이상 감지 경고
  • 업체 정보 뷰 — 계약 업체 연락처·계약서 첨부
  • 채팅 — 플래너와 실시간 문서 공유 메시지
플래너 전용 대시보드 플래너 전용
  • 담당 커플 목록 — 커플별 준비 상태 한눈에 확인
  • 비용 명세서 입력·수정 — 항목별 금액, 계약외 옵션 플래그
  • 일정 등록 — 준비 단계별 날짜 등록, 커플과 공유
  • 업체 연계 관리 — 연계 업체 등록, 추천 배지 관리

시스템 아키텍처 설계서 작성

실제 시스템에서 필요한 기능을 도출하고, 각 레이어에 적합한 기술 스택을 결정했습니다. 무료 티어 우선 원칙과 서버리스 아키텍처를 기반으로 선택했습니다.

01
Frontend React Native (Expo)

iOS · Android 크로스플랫폼. Expo Router 기반 파일 시스템 라우팅. 커플/플래너 역할별 레이아웃 분기.

React Native Expo Expo Router TypeScript
02
Backend API FastAPI

Python 기반 REST API 서버. AI 기능(Claude API 호출, 시세 이상 감지 엔진) 처리. Pydantic 스키마 유효성 검사.

FastAPI Python Pydantic Uvicorn
03
Database Supabase (PostgreSQL)

무료 티어 제공. 인증(Auth), 실시간 구독(Realtime), Row-Level Security 내장. AWS RDS 대비 초기 비용 없음.

Supabase PostgreSQL Realtime RLS
04
Cache / Session Upstash (Redis) — 검토 중

AI 응답 캐시(동일 입력 1시간 재사용), Rate Limiting, 세션 관리에 필요. Supabase가 Redis를 대체할 수 없어 서버리스 Redis인 Upstash 사용을 검토 중. 이후 추가 조사 후 확정 예정.

Upstash Redis 검토 중
05
AI Layer Anthropic Claude API

예산 견적 자연어 해설 생성(FR-052), 업체 추천 이유 생성(FR-057). claude-sonnet 모델, 스트리밍 응답 적용. 월 30만원 이하 비용 관리 목표.

Claude API Streaming RAG
06
UI Prototype Stitch

기초적인 앱 UI 프로토타입 구성. Rose · Sage 컬러 시스템 초안 확인. 실제 구현 전 화면 흐름 검증 용도.

Stitch UI Prototype

03 — Issues & Solutions

이슈 상황 및 해결 방법

Issue 01 기술 스택 선택 — 데이터베이스
상황
시스템 아키텍처 설계서 작성 시 어떤 기술 스택을 사용할지 결정하기 어려웠습니다. 특히 데이터베이스 선택에서 AWS RDS vs Supabase 사이에서 고민이 있었습니다.
해결
초기 개발 단계에서는 비용 부담이 없는 Supabase 무료 티어로 결정. PostgreSQL 기반이라 추후 AWS 마이그레이션도 용이하며, 인증·Realtime·RLS가 내장되어 있어 초기 개발 속도에 유리합니다.
Issue 02 — 미해결 AI 캐시 / Rate Limiting / 세션 관리 레이어
상황
AI 응답 캐시, Rate Limiting, 세션 관리를 위해 Redis가 필요한데, Supabase가 이를 대체할 수 없다는 것을 검색을 통해 확인했습니다. AWS ElastiCache는 비용 문제가 있어 대안이 필요했습니다.
현재 방향
Upstash(서버리스 Redis)를 유력 후보로 검토 중. 서버리스 방식이라 요청당 과금 구조로 초기 비용이 낮습니다. 무료 티어(하루 10,000 커맨드) 확인 후 채택 여부를 결정할 예정입니다.
다음 액션
Upstash 공식 문서 및 FastAPI 연동 레퍼런스 추가 조사. Sprint 1(1~2주차) 내 확정 목표.

05 — Retrospective

느낀점

이번에 개발해야 하는 것이 그룹웨어다 보니 만들어야 하는 페이지나 기능이 꽤 많습니다. 생각보다 개발 시간이 부족할 것 같아 내일부터 열심히 개발 진도를 나가야 할 것 같습니다.

Keep 잘한 것
  • 솔루션 분석을 통해 시장 공백을 명확히 도출하고, 근거 있는 주제 선정을 진행한 것
  • 화면을 4개 영역으로 구분해 기능 범위를 구체적으로 정의한 것
  • 기술 스택을 선택할 때 비용·확장성 기준을 함께 고려한 것
Problem 아쉬운 것
  • 폴더 구조 작성이 오늘 완료되지 못했습니다. 내일 환경설정 전에 먼저 확정이 필요합니다.
  • Upstash(Redis) 관련 기술 검토가 미완료 상태로 아키텍처 일부가 확정되지 않았습니다.
Try 개선할 것
  • 그룹웨어 특성상 기능 범위가 넓으므로, 매일 스프린트 목표를 명확히 설정하고 우선순위를 지켜 진행합니다.
  • 기술 결정 사항은 이유와 함께 문서화하여 팀원 간 컨텍스트를 공유합니다.
