Section 01 — Vendor System

Vendor 시스템 구축 과정 트러블슈팅

#01   Property ReferenceError — 데이터 소스 전환 오류

Property 'hallData' (또는 'ALL_VENDORS') doesn't exist

원인 import 제거 후 내부 변수 참조 잔존
상황
로컬 JSON 파일을 삭제하고 Supabase API로 교체하는 과정에서 화면 렌더링 불가 및 에러 발생.
원인
import hallData from ... 구문은 삭제했으나, useMemo 또는 useEffect 내부에서 해당 변수를 여전히 참조하여 데이터 가공을 시도.

해결 방법

  • 참조 제거: 소스 코드 전반에서 로컬 변수(hallData, studioData 등)를 참조하는 Spread 연산자 및 매핑 로직 전량 제거.
  • 상태값 기반 전환: useState로 선언된 vendors 배열을 단일 데이터 원천(Single Source of Truth)으로 지정, 서버 데이터 수신 후 setVendors() 호출로 구조 변경.

#02   Unmatched Route — 경로 인식 불가

Unmatched Route — The route '/(couple)/vendors/123' could not be found.

원인 1 그룹 폴더 오해 — (couple)을 URL에 직접 기입
상황
업체 상세 페이지 이동 시 Expo Router가 경로를 찾지 못하고 에러 화면 출력.
원인
(couple)과 같은 괄호 폴더는 URL 경로에 포함되지 않음에도 불구하고 주소에 직접 기입.
원인 2 동적 파라미터 누락 — category 미전달
원인
상세 페이지 호출 시 어떤 테이블에서 데이터를 가져와야 할지 알려주는 category 파라미터가 미전달되어 데이터 로드 실패.
수정 전 ❌ 
router.push(`/(couple)/vendors/${id}`);
수정 후 ✅ 
router.push(`/vendors/${id}?category=${activeCategory}`);

(couple)을 경로에서 제거하고, category를 쿼리 스트링으로 전달하여 상세 페이지가 즉시 특정 테이블을 조회할 수 있도록 수정.

#03   로딩 상태와 '결과 없음' 메시지 충돌 — UX 오류

원인 loading 상태 미분리 — 빈 배열 초기값 오인식
상황
앱 구동 직후 또는 카테고리 변경 시 데이터 로드 전 "해당하는 업체가 없습니다" 메시지가 노출되어 사용자 혼란 유발.
원인
FlatList의 ListEmptyComponent가 데이터 배열 길이(vendors.length === 0)만 체크하여, 초기 빈 배열 상태인 로딩 중에도 '데이터 없음'으로 간주. 
const renderEmpty = () => {
  if (loading) return <ActivityIndicator />;  // 로딩 중 스피너
  return <Text>해당하는 업체가 없습니다.</Text>;  // 로딩 완료 후 데이터 없을 때만
};

#04   이미지 렌더링 실패 — 프로토콜 미지정

원인 프로토콜 생략 URL — 모바일 환경 미인식
상황
일부 업체 이미지가 화면에 출력되지 않고 에러(엑스박스)로 표시.
원인
DB에 저장된 일부 이미지 URL이 //cdn.example.com/... 형태로 프로토콜(http:, https:)이 생략된 상대 경로로 저장되어 모바일 환경에서 인식 불가. 
// 이미지 source 지정 시 startsWith('//') 조건으로 https: 강제 접두
uri: url.startsWith('//') ? `https:${url}` : url

Section 1 핵심 레슨: ① Expo Router 그룹 폴더(())는 URL에서 생략됨을 반드시 숙지. ② 서버 데이터 처리 시 loading 상태를 반드시 함께 관리하여 사용자 피드백 제공. ③ DB 전환 시 기존 JSON 키와 DB 컬럼명의 일치 여부 검수 필수.

Section 02 — Consultation Notification

상담 신청 알림 시스템 트러블슈팅

사용자가 플래너에게 상담 신청 시, 플래너 계정으로 실시간 알림을 전송하는 기능 구축 과정에서 발생한 트러블슈팅입니다. 단순 프로필 정보가 아닌 실제 유저 UUID를 찾아 연결하는 것이 핵심이었습니다.

최종 데이터 흐름도

① 사용자: 상담 신청 버튼 클릭
② App: auth.getUser()로 신청자(sender_id) 확보
③ App: user_profiles Join으로 플래너의 실제 계정 UUID(user_id) 확보
④ Supabase: notifications 테이블에 발신자, 수신자, 알림 내용 저장
⑤ 결과: 플래너가 누가, 언제 신청했는지 확인하고 sender_id로 즉시 채팅 시작 가능

#05   알림 수신자(user_id) 식별 오류

FK 위반 — wedding_planners.id를 notifications.user_id에 삽입 시 에러 발생

원인 프로필 ID vs 계정 UUID 혼동
원인
wedding_planners.id는 플래너 프로필 ID이며, notifications.user_id는 실제 계정(auth.users)의 UUID여야 함.
해결
user_profiles 테이블을 Join하여 플래너 프로필에 연결된 실제 계정 UUID를 조회하도록 쿼리 수정. 
// wedding_planners와 user_profiles를 Join하여 실제 UUID 확보
const { data } = await supabase
  .from('wedding_planners')
  .select('*, user_profiles!planner_id(id)')
  .eq('name', plannerName)
  .single();
// 수신자 UUID: data.user_profiles[0].id

#06   외래 키 제약 조건 위반 — Type Error

23503 Foreign Key Constraint — notifications.type 컬럼 FK 위반

원인 notification_types에 없는 'info' 타입 사용
원인
notifications.type 컬럼이 notification_types 테이블을 참조하는데, 코드에서 전달한 'info'라는 값이 해당 테이블에 미등록.
해결
notification_types 테이블을 조회한 후 실제 등록된 값인 'info_consultation'(또는 'notice')으로 코드 수정.

#07   발신자 정보(sender_id) 누락 및 채팅 연동 불가

원인 초기 테이블 설계에 발신자 컬럼 누락
현상
알림은 전송되지만 플래너가 발신자를 알 수 없어 채팅 연결이 불가능.
해결
DB에 sender_id 컬럼(UUID 타입, auth.users 참조) 추가. 알림 전송 시 supabase.auth.getUser()로 현재 로그인 유저 ID를 함께 저장하도록 로직 수정.

#08   'property user doesn't exist' 런타임 에러

property 'user' doesn't exist — 유저 ID 조회 과정에서 앱 강제 종료 또는 에러 Alert 발생

원인 auth.getUser() 응답 객체 비구조화 할당 오류
원인
Supabase Auth 응답 객체의 비구조화 할당 오류. data 객체를 거치지 않고 user에 직접 접근 시도.
해결
Optional Chaining(?.)을 활용한 안전한 객체 접근 방식으로 개선.
수정 전 ❌ 
const { user } = await supabase.auth.getUser();
const myId = user.id;  // TypeError 발생
수정 후 ✅ 
const authRes = await supabase.auth.getUser();
const myId = authRes.data?.user?.id;  // Optional Chaining 적용

Section 2 핵심 레슨: ① '프로필 ID'와 '계정 UUID'를 엄격히 구분하여 개발. ② 컬럼 추가 후 클라이언트 코드 반영 여부 및 메트로 서버 재시작 확인. ③ Supabase API 호출 시 null 가능성을 고려한 Optional Chaining 활용.

Section 03 — Expo Router

Expo Router 라우팅 및 역할 기반 접근 제어 트러블슈팅

#09   TypeError: Cannot read property 'filter' of undefined

TypeError: Cannot read property 'filter' of undefined — TabRouter.js, useNavigationBuilder.js

앱 실행 시 흰 화면이 뜨거나 Red Box 에러 발생. 호출 스택이 TabRouter.js, useNavigationBuilder.js를 가리키며 Stack → Tabs 레이아웃 변경 직후 주로 발생합니다.

원인 1 Tabs.Screen name과 실제 파일명 불일치
원인
_layout.jsx 내의 <Tabs.Screen name="..." />에 명시된 name과 실제 폴더 내의 물리적 파일이 존재하지 않거나 이름이 다름.
원인 2 Tabs 설정에서 라우터 경로 필터링 실패
원인
Expo Router는 폴더 내의 모든 파일을 라우트로 간주하는데, Tabs 설정에서 이를 제대로 필터링하지 못할 때 발생.

해결 방법

  • 파일명 전수 조사: Tabs.Screen의 name 속성과 실제 .jsx 파일명의 대소문자, 특수문자(언더바/하이픈) 일치 여부 확인.
  • 빈 파일 생성: 미구현 기능(chat.jsx 등)도 Tabs.Screen에 등록되어 있다면 빈 컴포넌트 파일이라도 생성 필요.
  • 캐시 강제 삭제: 파일 구조 변경 후 메트로 번들러 캐시에 즉각 반영되지 않을 수 있으므로 캐시를 지우고 재시작. 
npx expo start -c  // 캐시 강제 삭제 후 재시작

#10   WARN: No route named "X" exists in nested children

터미널/콘솔에 노란색 경고창 발생. 특정 탭 아이콘이 미표시되거나 클릭 시 무반응인 증상입니다.

원인 _layout 등록 name에 해당하는 파일 누락
원인
_layout.jsx에 코드는 있으나, Expo Router가 인식하는 실제 파일 목록(Nested Children)에 해당 파일이 누락.
해결
경고 메시지의 nested children: [...] 리스트를 확인한 후, 목록에 없는 파일명은 _layout.jsx에서 삭제하거나 실제 파일을 생성.

#11   역할 기반 라우팅(Role-based Routing) 실패

플래너로 로그인했는데 커플 페이지가 뜨거나, 인증 후에도 로그인 화면에 멈춰 있는 증상입니다.

원인 1 세션 반환 속도보다 렌더링이 빨라 잘못된 리다이렉션
원인
app/_layout.jsx의 AuthGate 로직에서 Supabase 세션 반환 속도보다 렌더링이 빨라 잘못된 리다이렉션 발생.
해결
세션과 프로필 로드 완료 전까지 ActivityIndicator를 표시하여 잘못된 리다이렉션 방지.
원인 2 role 컬럼 대소문자 불일치
원인
user_profiles 테이블의 role 컬럼 값 대소문자 불일치 (예: 'Planner' vs 'planner').
해결
console.log(profile.role)로 실제 DB 문자열을 확인하고 조건문과 일치시키기.

#12   알림 데이터 연동 오류 — Context API

알림 목록이 비어있거나, 읽음 처리(markAsRead) 후 화면에 즉시 미반영되는 증상입니다.

원인 1 Supabase Realtime 구독 누락
원인
Supabase Realtime 설정 미활성화 또는 useEffect 내 구독 로직 오류.
해결
supabase.channel().subscribe() 정상 연결 여부를 네트워크 탭에서 확인.
원인 2 is_read(snake_case) vs isRead(camelCase) 필드명 불일치
원인
DB 컬럼명(is_read, snake_case)과 코드 변수명(isRead, camelCase) 불일치로 매핑 오류 발생.
해결
fetchNotifications에서 DB 데이터를 가져올 때 UI 형식으로 변환하는 매핑 계층 추가. 
// DB snake_case → UI camelCase 변환
const mapped = data.map(n => ({ ...n, isRead: n.is_read }));

Best Practices: ① 파일 생성 후 _layout.jsx에 등록하는 순서를 습관화. ② 탭 바 미표시 상세 페이지에 options={{ href: null }} 설정으로 라우터 충돌 방지. ③ Supabase 연동 문제 발생 시 .env 파일의 URL 및 API Key 우선 확인.

04 — Core Principles

종합 핵심 원칙

원칙 01 단일 데이터 원천 (Single Source of Truth)
useState를 통한 단일 데이터 원천 유지. 로컬 변수 참조를 제거하고 서버에서 받아온 데이터를 상태값으로 일원화.
원칙 02 로딩 상태 관리
비동기 처리 시 loading 플래그 필수. ActivityIndicator로 사용자 피드백을 제공하고 로딩 상태와 결과 없음 상태를 반드시 분리.
원칙 03 ID 체계 명확화
프로필 ID와 계정 UUID(auth.users)를 엄격히 구분. FK 참조 테이블을 사전에 확인하여 위반 에러 방지.
원칙 04 방어적 프로그래밍
Optional Chaining(?.) 적극 활용. null / undefined 방어 로직을 외부 API 호출 모든 지점에 상시 적용.
원칙 05 라우팅 규칙 준수
