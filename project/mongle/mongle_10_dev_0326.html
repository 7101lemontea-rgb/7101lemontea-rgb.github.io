Overview

프로젝트 개요

Mongle은 예비 부부와 웨딩 플래너를 연결하는 웨딩 앱입니다. 본 문서는 2026년 3월 26일 작업 내용을 기록하며, 플래너 전용 기능 신설, 알림 시스템 구현, 플래너 상세 화면 상담 신청 모달 구현에 관한 상세 명세를 담고 있습니다.

프레임워크
React Native (Expo SDK)
라우터
Expo Router (파일 기반 라우팅)
아이콘
@expo/vector-icons (Ionicons)
상태 관리
React Context API + useState
예정 백엔드
Supabase (PostgreSQL)
디자인 스타일

02 — dashboard.jsx

플래너 전용 대시보드

플래너가 로그인 후 첫 번째로 보는 홈 화면으로, 업무 전반을 한눈에 파악할 수 있도록 5개 카드 섹션으로 구성했습니다.

카드 섹션 구성 내용
플래너 정보 카드 플래너 이름·직함 표시, 오늘의 할 일 건수 요약 배지
담당 커플 현황 담당 커플 수 / 이번 달 결혼식 수 KPI, 임박 커플 D-day 프로그레스 바 (최대 4개)
오늘 할 일 (Todo) 마감 임박 항목 강조, 체크박스 완료 토글, 최대 5개 표시, 전체 보기 → planner_todo_list
협력 업체 현황 카테고리별 섹션 (업체 없는 카테고리 숨김), 최대 3카테고리 × 3업체 표시, 전체 보기 → wedding_vendor_partners
비용 현황 수금 총액·미수금·미지급 업체 KPI, 미수금 커플 프로그레스 바, 미지급 업체 총액 알림, 상세 보기 → planner_budget

라우팅 구조

진입 지점 라우팅 코드

04 — Notifications

알림 기능 개발 명세

알림 관련 기능 전체를 독립 화면으로 분리하고, Context API를 통해 전역 알림 상태를 관리합니다.

4-1. notifications.jsx — 화면 구성 요소

UI 요소 설명
헤더 뒤로 가기 버튼 + '알림' 타이틀 + '모두 읽음' 버튼 (읽지 않은 알림 있을 때만 표시)
읽지 않은 알림 섹션 섹션 레이블, 제목 굵게 표시, 빨간 점 뱃지
읽은 알림 섹션 섹션 레이블, 흐린 스타일로 구분
빈 상태 알림이 없을 때 아이콘 + 안내 문구 표시
상세 모달 하단 슬라이드 업, 아이콘·제목·시간·본문·삭제 버튼 포함

4-2. NotificationContext 설계

_layout.jsx에서 NotificationProvider로 전체 화면을 감싸면, Provider 안의 어느 화면에서든 useNotifications() 훅으로 알림 데이터를 바로 꺼내 쓸 수 있습니다. props drilling 없이 전역 접근이 가능합니다. 

// context/NotificationContext.jsx 핵심 인터페이스
const { notifications, unreadCount, setUserId,
        markAsRead, markAllAsRead, deleteNotification
} = useNotifications();

// 로그인 성공 후 userId 전달 → 해당 유저 알림 자동 fetch
setUserId(user.id);

4-3. DB 테이블 설계 — notifications

컬럼명 타입 설명
id BIGINT PK 알림 고유 ID (자동 증가)
user_id BIGINT FK 수신자 ID — users 테이블 참조
type VARCHAR(50) 알림 종류 (notification_types 참조)
title VARCHAR(200) 알림 제목
body TEXT 알림 본문
icon VARCHAR(50) Ionicons 아이콘 이름
icon_color / icon_bg VARCHAR(10) 아이콘 색상 및 배경 HEX 코드
is_read BOOLEAN 읽음 여부 (기본값: FALSE)
created_at DATETIME 생성 일시 (기본값: NOW())
ref_type VARCHAR(50) 연결 대상 타입: couple / todo / vendor / finance
ref_id BIGINT 연결 대상 레코드 ID (딥링크 이동에 활용)

4-4. API 엔드포인트

메서드 엔드포인트 설명
GET /api/notifications?userId={id} 사용자 알림 목록 조회 (unreadCount 포함)
PATCH /api/notifications/:id/read 특정 알림 읽음 처리
PATCH /api/notifications/read-all 전체 알림 일괄 읽음 처리
DELETE /api/notifications/:id 특정 알림 삭제

4-5. 알림 자동 생성 정책

트리거 조건 알림 타입 생성 주체 비고
할 일 마감일 당일 오전 9시 todo_deadline cron 담당 플래너 대상
잔금 납부일 10일 전 payment_due cron 담당 플래너 대상
결혼식 D-30, D-14, D-7, D-1 dday_reminder cron 담당 플래너 대상
할 일 완료 처리 시 todo_completed API 서버 즉시 발송
새 커플 등록 시 couple_added API 서버 즉시 발송
05 — planners/[id].jsx

플래너 상세 화면 — 상담 신청 모달

파일 경로: app/(couple)/planners/[id].jsx. Expo Router 동적 라우팅을 사용하며, 플래너 이름을 URL 파라미터로 받아 planners.json에서 해당 데이터를 조회합니다.

5-1. State 구성

State 변수 초기값 타입 용도
modalVisible false Boolean 모달 표시 여부 제어
consultYear '' String 희망 날짜 — 년도 (4자리 숫자)
consultMonth '' String 희망 날짜 — 월 (최대 2자리)
consultDay '' String 희망 날짜 — 일 (최대 2자리)
consultAmPm '오전' String 희망 시간 — 오전/오후 토글
consultHour '' String 희망 시간 — 시간 숫자 입력
consultBudget '' String 예산 (만원 단위 숫자)
submitting false Boolean API 호출 중 로딩 상태

5-2. 입력 필드 상세

필드 구현 명세
날짜 — 년도 TextInput, keyboardType='numeric', maxLength=4, 뒤에 '년' 텍스트
날짜 — 월 TextInput, keyboardType='numeric', maxLength=2, 뒤에 '월' 텍스트
날짜 — 일 TextInput, keyboardType='numeric', maxLength=2, 뒤에 '일' 텍스트
날짜 레이아웃 세 필드가 dateRow로 가로 배치. 전송 시 padStart(2,'0') 포맷 통일 → YYYY-MM-DD 조합
시간 — 오전/오후 두 버튼 붙은 ampmToggle UI. 선택 쪽에 어두운 배경 적용
시간 — 시간 숫자 TextInput, keyboardType='numeric', maxLength=2. '오전 2시' 형식으로 조합
예산 TextInput, keyboardType='numeric'. 만원 단위 숫자 입력

5-3. 유효성 검사 및 전송 처리

submitBtn 탭 시 아래 조건을 모두 확인하며, 하나라도 비어 있으면 Alert를 표시하고 전송을 중단합니다.

  • consultYear, consultMonth, consultDay 모두 입력 여부 확인
  • consultHour 입력 여부 확인
  • consultBudget 입력 여부 확인

전송 처리는 handleConsultSubmit 함수가 sendNotificationToPlanner를 호출하며, 현재는 console.log + 800ms 딜레이로 구현되어 있습니다. 실제 서비스 연동 시 아래 코드로 교체합니다. 

await fetch('https://your-api.com/notify', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ plannerName, date, time, budget })
});

5-4. 네비게이션

항목 설명
진입 경로 커플 홈 플래너 목록에서 카드 탭 → router.push('/(couple)/planners/' + planner.name)
뒤로 가기 (헤더 버튼) router.replace('/(couple)') — 홈 화면으로 이동
뒤로 가기 (Android 물리) BackHandler useEffect 등록 → router.replace('/(couple)')
06 — Supabase DB

Supabase DB 설계

모든 테이블은 planner_id 외래키로 플래너와 연결됩니다. couples 테이블이 수금·일정·업체지출의 중심이 됩니다.

테이블 관계 구조

테이블 연결 관계 및 역할
planners 최상위 — planner_id 기준으로 모든 데이터 소유
couples planner_id → planners
couple_payments couple_id → couples (계약금·중도금·잔금 납부 일정)
couple_schedules couple_id → couples (드레스·스튜디오·예식 등 일정)
vendors planner_id → planners (협력 업체 정보)
vendor_costs vendor_id → vendors, couple_id → couples (업체 지출 내역)
todos planner_id → planners, couple_id → couples (선택)
notifications user_id → users, ref_id → 대상 테이블 (알림)

주요 Supabase 조회 예시

// 커플 목록 조회 (납부 일정 + 일정 포함)
const { data } = await supabase
  .from('couples')
  .select('*, couple_payments(*), couple_schedules(*)')
  .eq('planner_id', plannerId)
  .order('wedding_date', { ascending: true });

// 할 일 목록 조회 (커플명 포함)
const { data } = await supabase
  .from('todos')
  .select('*, couples(groom_name, bride_name)')
  .eq('planner_id', plannerId)
  .order('due_date', { ascending: true });

// 업체 지출 조회 (업체·커플명 포함)
const { data } = await supabase
  .from('vendor_costs')
  .select('*, vendors(name, type), couples(groom_name, bride_name)');
