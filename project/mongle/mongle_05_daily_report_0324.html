라우팅 구조 구현 (Expo Router)

루트 레이아웃 app/_layout.jsx를 구현하고, Expo Router를 통해 각 화면 (timeline.jsx, login.jsx 등) 간 이동이 가능하도록 구성했습니다. 기존 App.js 방식을 app/_layout.jsx로 완전히 대체했습니다.

L1
app/_layout.jsx 루트 레이아웃

앱 진입점. 인증 상태 확인 → 미로그인 시 (auth)/login으로 리다이렉트(인증 가드). 로그인 시 역할(커플/플래너)에 따라 레이아웃 분기. 기존 App.js를 완전히 대체.

L2
(auth)/_layout.jsx 인증 스택

로그인·회원가입 화면 스택 네비게이터. 인증 완료 후 커플/플래너 레이아웃으로 이동. login.jsx, register.jsx 포함.

L3
(couple)/_layout.jsx 커플 탭 네비게이터

커플 전용 하단 탭 네비게이터. 탭 4개 — 홈(index.jsx) · 일정(timeline.jsx) · 비용(budget.jsx) · 채팅(chat.jsx). 각 파일에 임시 컴포넌트 배치 완료.

L4
(planner)/_layout.jsx 플래너 탭 네비게이터

플래너 전용 레이아웃 뼈대. 담당 커플 목록·비용 입력·일정 관리 탭 구조 초안. 상세 구현은 이후 스프린트에서 진행.

timeline.jsx UI 초안 확정

(couple)/timeline.jsx에 포함될 D-day 배너, 달력, 타임라인의 UI를 Figma로 작성하고 대략적인 레이아웃을 확정했습니다.

03 — Issues & Solutions

이슈 상황 및 해결 방법

a. npm install 의존성 충돌

npm error ERESOLVE could not resolve
npm error Conflicting peer dependency: react@19.2.4

원인
react@19.1.0과 react-dom@19.2.4의 peer dependency 버전이 맞지 않아 발생. 패키지가 서로 다른 React 버전을 요구하면서 npm이 의존성 트리 해석에 실패했습니다.
해결 방법
npm install --legacy-peer-deps 옵션 사용. peer dependency 충돌을 무시하고 설치를 진행합니다.
주의사항
--legacy-peer-deps는 임시 해결책입니다. 이후 의존성 버전을 명시적으로 맞추거나, Expo SDK 버전과 호환되는 패키지 버전으로 고정하는 것을 권장합니다. 
# 의존성 충돌 무시하고 설치
npm install --legacy-peer-deps

b. Route missing default export

WARN Route "./(couple)/chat.jsx" is missing the required default export.
WARN Ensure a React component is exported as default.

원인
Expo Router는 app/ 폴더 안의 모든 .jsx / .tsx 파일을 자동으로 라우트로 인식합니다. 파일이 존재하지만 export default가 없거나 파일이 비어 있으면 이 경고가 발생합니다.
해결 방법
아직 구현하지 않은 파일에 임시 컴포넌트를 넣어 경고를 제거했습니다. 빈 파일을 그대로 두지 않는 것이 핵심입니다. 
// 임시 컴포넌트 — 구현 전 placeholder
import { View, Text } from 'react-native';

export default function ChatScreen() {
  return (
    <View><Text>준비 중</Text></View>
  );
}

c. 화면에 "layout" 글씨만 보임

증상
앱을 실행하면 메인 화면 대신 "layout" 또는 "_layout" 텍스트만 화면에 표시되는 문제가 발생했습니다.
원인
_layout.jsx 파일 중 하나가 올바른 네비게이터(<Stack> 또는 <Tabs>)를 반환하지 않고 텍스트를 렌더링하고 있었습니다. 특히 (auth)/_layout.jsx가 임시 컴포넌트 상태로 남아있을 때 발생하는 문제였습니다.
해결 방법
해당 파일을 올바른 네비게이터 컴포넌트로 수정했습니다. _layout.jsx는 반드시 <Stack> 또는 <Tabs> 같은 네비게이터를 반환해야 합니다. 
// ❌ 잘못된 _layout.jsx — 텍스트 렌더링
export default function AuthLayout() {
  return <Text>layout</Text>; // ← 문제 원인
}

// ✅ 올바른 _layout.jsx — Stack 네비게이터 반환
import { Stack } from 'expo-router';

export default function AuthLayout() {
  return <Stack screenOptions={{ headerShown: false }} />;
}

핵심 규칙: Expo Router에서 _layout.jsx는 반드시 <Stack>, <Tabs>, <Drawer> 중 하나의 네비게이터를 반환해야 합니다. 일반 컴포넌트나 텍스트를 반환하면 화면이 올바르게 렌더링되지 않습니다.

05 — Retrospective

느낀점

Expo 환경설정과 Git 설정이 생각보다 복잡했습니다. 팀장님이 설정하신 것을 바탕으로 어떻게 Git을 활용해야 하는지 배울 수 있었습니다. Expo Router를 활용하여 폴더 안의 파일들을 오갈 수 있게 설정하는 것이 초기 시스템 설계의 핵심이었는데, 배경지식이 부족하여 앱 초기 화면 구성 방법을 이해하는 데 시간이 걸렸습니다.

Route missing default export와 관련된 오류로 아직 구현하지 않은 파일에 임시 컴포넌트를 넣어 우선 해결했지만, 이후 앱을 실행하면 메인 화면 대신 "layout" 또는 "_layout" 텍스트만 화면에 표시되는 문제가 이어졌습니다. 이와 관련된 오류들을 해결하느라 꽤 고생했지만, 앞으로는 왜 발생하는지 알게 되었습니다.

Keep 잘한 것
  • 팀원 전원의 환경설정을 하루 안에 통일해 공동 개발 기반을 마련한 것
  • Figma에서 디자인 시스템을 먼저 구축하고 코드로 연결한 것. 이후 UI 작업의 일관성이 높아질 것입니다.
  • 이슈가 발생했을 때 원인을 파악하고 문서화한 것. 같은 오류가 재발했을 때 빠르게 해결할 수 있습니다.
Problem 아쉬운 것
  • Expo Router 구조를 처음 접해 초기 설정에 예상보다 많은 시간을 소비했습니다. 공식 문서를 미리 읽었다면 더 효율적으로 진행할 수 있었을 것입니다.
Try 개선할 것
  • 새로운 기술 스택(expo-router 등)을 사용할 때 공식 문서와 기본 예제를 먼저 확인하는 습관을 만들겠습니다.
  • _layout.jsx의 역할 — 반드시 네비게이터를 반환해야 한다는 규칙 — 을 팀원 전체와 공유합니다.
  • 미완성 파일을 생성할 때 임시 컴포넌트 템플릿을 미리 정해두어 "Route missing" 경고가 발생하지 않도록 합니다.
