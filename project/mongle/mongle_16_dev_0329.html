Part 01 — Vendor Data System

Vendor 데이터 시스템 전환 및 UX 최적화

정적 로컬 파일(.json) 기반 데이터를 실시간 클라우드 DB인 Supabase로 전환하여 데이터 관리의 유연성을 확보하고, 비동기 환경에서 발생할 수 있는 로딩 지연 및 라우팅 에러를 해결했습니다.

1-1. 아키텍처 변경

기존 방식 Static

모든 JSON 데이터를 앱 로드 시점에 import하여 메모리에 상주시킴.

변경 방식 Dynamic

supabase.from('table_name')을 사용하여 탭 전환 및 페이지 진입 시점에 필요한 데이터를 비동기(async/await)로 호출.

카테고리 ID에 따라 조회할 DB 테이블을 동적으로 결정하는 매핑 객체(TABLE_MAP)를 도입했습니다.

카테고리 ID Supabase 테이블명 주요 데이터 구조
hall hall_vendors basic_info, content, pricing, details (JSONB)
studio studio_vendors 상동
dress dress_vendors 상동
makeup makeup_vendors 상동
video_snap video_snap_vendors 상동
package package_vendors 상동

1-2. 기술 구현 및 UX 최적화

데이터를 불러오는 중(loading: true)과 완료 후(loading: false)를 명확히 구분하여 사용자 혼란을 방지합니다. FlatList의 ListEmptyComponent를 활용한 조건부 렌더링으로 구현했습니다. 

const renderEmpty = () => {
  if (loading) {

        <Text style={styles.loadingText}>업체 리스트를 불러오고 있습니다...</Text>
      </View>
    );
  }
  return (
    <View style={styles.emptyContainer}>
      <Text style={styles.emptyText}>해당하는 업체가 없습니다.</Text>
    </View>

Unmatched Route 에러를 해결하고 상세 페이지의 쿼리 효율을 높이기 위해 주소 전달 방식을 수정했습니다.

수정 전 ❌ 
router.push(`/vendors/${id}`)
수정 후 ✅ 
router.push(`/vendors/${id}?category=${activeCategory}`)

Expo Router의 그룹 폴더 (couple)을 URL 경로에서 제외하고, 상세 페이지에서 즉시 특정 테이블을 조회할 수 있도록 category를 쿼리 스트링으로 전달.

1-3. 트러블슈팅

Bug 01 ReferenceError — ALL_VENDORS 등 변수 참조 오류
원인
로컬 JSON import 제거 후 기존 변수명이 남아 있어 에러 발생.
해결
useState 및 useEffect를 통해 서버에서 받아온 데이터를 상태값으로 관리하도록 전면 수정.
Bug 02 경로 인식 오류 — (couple) 포함 및 파일명 불일치
원인
주소에 (couple)을 포함하거나 파일명([id].js)과 호출 주소가 불일치.
해결
실제 파일 시스템 경로와 매핑되는 절대 경로를 명시하고 pathname과 params 구조를 정돈.

1-4. 향후 주의사항 및 확장 가이드

  • 데이터 구조 동기화: Supabase 테이블 내 JSONB 컬럼(basic_info 등)의 키 값이 기존 JSON 구조와 100% 일치해야 렌더링 에러를 방지할 수 있음.
  • 이미지 전처리: 썸네일 URL이 //로 시작하는 경우 https:를 붙여주는 전처리 로직을 공통 유틸리티로 관리 권장.
  • 방어 로직 유지: 상세 페이지 진입 시 category 파라미터가 없을 경우를 대비한 searchAllTables 로직을 유지하여 데이터 유실 가능성 차단.
Part 02 — Consultation Notification

상담 신청 및 알림 시스템

사용자가 플래너 상세 페이지에서 상담을 신청했을 때, 해당 플래너의 계정으로 실시간 알림을 전송하는 시스템입니다. 단순 프로필 정보가 아닌 실제 유저 UUID를 찾아 연결하는 것이 핵심입니다.

2-1. DB 구조

세 개의 테이블이 연동되어 알림을 처리합니다.

wedding_planners

플래너의 영업용 프로필 (이름, 경력, 지역 등).

user_profiles

서비스의 통합 계정 정보. planner_id 컬럼을 통해 wedding_planners.id와 연결됨.

notifications

user_id는 수신자의 UUID, sender_id는 발신자의 UUID를 저장.

notifications 테이블 상세

컬럼명 타입 설명
id bigint Primary Key (PK)
user_id uuid 수신자 — 플래너의 실제 계정 UUID
sender_id uuid 발신자 — 상담 신청 유저의 UUID
type text 알림 유형 (notification_types 테이블 참조)
title text 알림 제목 (예: 새 상담 신청)
body text 알림 상세 내용 (날짜, 시간, 예산 등 포함)
ref_type text 클릭 시 이동할 페이지 유형 (예: consultation)

2-2. 핵심 구현 로직

플래너 프로필 정보만으로는 알림을 보낼 수 없으므로, user_profiles 테이블을 Join하여 실제 계정 ID를 가져옵니다. 

// 플래너의 실제 User ID 조회 (Join Query)
const { data } = await supabase
  .from('wedding_planners')
  .select(`
    *,
    user_profiles!planner_id ( id )
  `)
  .eq('name', id)
  .single();
// 수신자 ID 추출: data.user_profiles[0].id

// 발신자(나)의 ID 확보
const authRes = await supabase.auth.getUser();
const myId = authRes.data?.user?.id; // 16d9b12d-e4dc-...

2-3. 트러블슈팅

Bug 01 null value in column "user_id"
원인
플래너의 프로필 ID를 유저 ID 자리에 넣음.
해결
user_profiles Join을 통해 실제 UUID 확보.
Bug 02 Foreign Key Constraint (type)
원인
notification_types에 없는 'info' 타입 사용.
해결
테이블에 정의된 'notice' 또는 'info_consultation'으로 변경.
Bug 03 Property user doesn't exist
원인
auth.getUser() 응답 구조를 오해.
해결
res.data.user.id 경로로 안전하게 접근하도록 수정.
Bug 04 Sender ID missing
원인
알림 테이블에 발신자 정보가 없음.
해결
sender_id 컬럼 추가 및 auth.getUser() 값 할당.

2-4. 향후 확장 계획

  • 실시간 채팅: 저장된 sender_id를 활용하여 플래너가 알림을 눌렀을 때 신청자와의 1:1 채팅방 자동 생성.
  • 상태 관리: 알림의 is_read 컬럼을 활용하여 읽음 처리 및 뱃지(Badge) 기능 구현.
Part 03 — Expo Router Architecture

Expo Router 구조 설계 및 플래너 탭 네비게이션 구현

기존의 단순한 Stack 구조에서 유저의 역할(Planner/Couple)에 따라 완전히 독립된 네비게이션 경험을 제공하기 위해 그룹화된 라우팅 구조로 변경했습니다.

3-1. 라우팅 아키텍처 재설계

Expo Router의 (group) 기능을 사용하여 URL 경로에는 영향을 주지 않으면서 논리적으로 화면을 분리했습니다.

(auth) 비로그인

비로그인 사용자 전용 — Login, SignUp 화면.

(planner) 플래너

플래너 전용 대시보드 및 고객 관리 시스템.

(couple) 예비부부

예비부부 전용 일정 및 비용 관리 시스템.

app/_layout.jsx의 AuthGate에서 로그인 세션과 user_profiles.role 값을 확인하여 앱 실행 즉시 해당 그룹으로 사용자를 이동시킵니다. 

로그인 세션 확인
  └─ role === 'planner'  →  /(planner)/dashboard
  └─ role === 'couple'   →  /(couple)/index

3-2. 플래너 전용 하단 탭 네비게이션

app/(planner)/_layout.jsx를 Tabs 레이아웃으로 구성하여 플래너가 멀티태스킹을 효율적으로 할 수 있도록 했습니다.

탭 메뉴 구성

메뉴명 파일 경로 (name) 아이콘 주요 기능
홈 dashboard home-outline 대시보드 확인, 신규 알림 요약
일정 planner_todo_list calendar-outline 전체 고객의 웨딩 스케줄 관리
비용 planner_budget wallet-outline 업체 정산 및 예산 관리
마이 mypage person-outline 플래너 프로필 수정 및 설정

플래너 폴더 내에 존재하지만 하단 탭 바에는 노출되지 않아야 하는 화면을 href: null로 제어했습니다. 이 설정이 없으면 모든 파일이 탭 메뉴로 생성되어 TypeError나 경고가 발생할 수 있습니다.

<Tabs.Screen name="wedding_vendor_partners"    options={{ href: null }} />  // 업체 관리
<Tabs.Screen name="notifications"             options={{ href: null }} />  // 알림 센터

Part 04 — Notification Feature

알림(Notification) 기능 개발

Supabase notifications 테이블과 React Native 알림 화면을 연동하는 작업입니다. DB 컬럼명(snake_case)과 컴포넌트 필드명(camelCase) 불일치, userId 미주입 문제를 해결하여 실제 알림 데이터가 화면에 표시되도록 수정했습니다.

파일 수정 여부 내용
notifications.jsx ✅ 수정 DB 필드 접근 헬퍼 함수 추가
NotificationContext.jsx — 변경 없음 다른 곳에서 참조 중 — 원본 유지
app/_layout.jsx ✅ 수정 로그인 세션에서 userId Context 주입

4-1. 문제 분석

DB 스키마 vs 컴포넌트 필드명 불일치

Supabase notifications 테이블의 컬럼명은 snake_case이지만 기존 notifications.jsx는 camelCase 필드명을 직접 참조하고 있어 모든 값이 undefined로 렌더링되었습니다.

DB 컬럼 (실제) 기존 코드 참조명 비고
is_read isRead 읽음 여부 — 섹션 분기 및 unread dot에 사용
icon_color iconColor 아이콘 색상 — null 가능
icon_bg iconBg 아이콘 배경색 — null 가능
icon icon null인 경우 기본 아이콘 필요
created_at time DB에는 time 컬럼 없음, 변환 필요

userId가 Context에 주입되지 않음

NotificationContext는 외부에서 setUserId()를 호출해야 fetchNotifications()가 실행되는 구조입니다. 그러나 app/_layout.jsx의 AuthGate에서 세션 확인 후 setUserId를 한 번도 호출하지 않아 알림 목록이 항상 빈 배열로 유지되었습니다. 

// NotificationContext.jsx — userId가 있어야만 실행됨
useEffect(() => {
  if (userId) {
    fetchNotifications(userId);  // userId null이면 실행 안 됨
  } else {
    setNotifications([]);         // 항상 빈 배열
  }
}, [userId]);

4-2. 수정 내용

notifications.jsx — DB 필드 접근 헬퍼 함수 추가

NotificationContext.jsx를 수정하지 않고, notifications.jsx 상단에 헬퍼 함수를 추가하여 DB 필드를 안전하게 읽도록 처리했습니다. 

// ── DB 필드 접근 헬퍼 ─────────────────────────────────
const ICON_BY_TYPE = {
  info_consultation: 'calendar-outline',
  info_wedding:      'heart-outline',
  info_message:      'chatbubble-outline',
  info_payment:      'card-outline',
};
function getIcon(n)      { return n.icon || ICON_BY_TYPE[n.type] || 'notifications-outline'; }
function getIconColor(n) { return n.icon_color ?? '#c97b6e'; }
function getIconBg(n)    { return n.icon_bg    ?? '#fdf4f3'; }
function getIsRead(n)    { return n.is_read; }
function getTime(n) {
  if (!n.created_at) return '';
  return new Date(n.created_at).toLocaleString('ko-KR', {
    month: 'long', day: 'numeric', hour: '2-digit', minute: '2-digit',
  });
}

헬퍼 함수 적용 후 변경 위치 요약입니다.

위치 변경 전 변경 후
섹션 분기 (읽지 않은 알림) n.isRead getIsRead(n)
상세 모달 아이콘 배경 selectedNotif.iconBg getIconBg(selectedNotif)
상세 모달 아이콘 selectedNotif.icon getIcon(selectedNotif)
상세 모달 아이콘 색상 selectedNotif.iconColor getIconColor(selectedNotif)
상세 모달 시간 selectedNotif.time getTime(selectedNotif)
NotifItem 읽음 여부 item.isRead getIsRead(item)
NotifItem 시간 item.time getTime(item)

app/_layout.jsx — AuthGate에 setUserId 주입

AuthGate에서 Supabase 세션 확인 직후 setUserId()를 호출하도록 수정했습니다. 로그아웃 시에는 setUserId(null)을 호출하여 알림 목록을 초기화합니다. 

function AuthGate() {
  const { setUserId } = useNotifications(); // 추가
  useEffect(() => {
    const initialize = async () => {
      const { data: { session } } = await supabase.auth.getSession();
      setSession(session);
      if (session) {
        const profile = await fetchUserRole(session.user.id);
        setRole(profile?.role || 'couple');
        setUserId(session.user.id); // 추가
      }
    };
    initialize();
    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      async (_event, session) => {
        setSession(session);
        if (session) {
          const profile = await fetchUserRole(session.user.id);
          setRole(profile?.role || 'couple');
          setUserId(session.user.id); // 추가
        } else {
          setRole(null);
          setUserId(null); // 추가 (로그아웃 시 알림 초기화)
        }
      }
    );
    return () => subscription.unsubscribe();
  }, [setUserId]); // dependency 추가
}

4-3. 최종 데이터 흐름

앱 시작 흐름
RootLayout: NotificationProvider로 감쌈
  └─ AuthGate: Supabase 세션 확인
       └─ setUserId(session.user.id)  ← 이번에 추가
            └─ NotificationContext: fetchNotifications(userId) 실행
                 └─ Supabase DB에서 notifications 조회
                      └─ notifications.jsx: 헬퍼 함수로 DB 필드 안전하게 렌더링

실시간 알림 흐름
Supabase Realtime 이벤트 (외부)
  └─ app/_layout.jsx: addRealtimeNotification(newNoti) 호출
       └─ NotificationContext: 상태에 새 알림 prepend
            └─ notifications.jsx: 자동 리렌더링

4-4. 트러블슈팅

Bug 01 알림 목록이 항상 비어 있음
원인
AuthGate에서 setUserId()를 호출하지 않아 fetchNotifications가 실행되지 않음.
해결
AuthGate에 setUserId 호출 추가 (초기화 + onAuthStateChange).
Bug 02 아이콘·색상·읽음 상태가 표시되지 않음
원인
DB는 snake_case 컬럼명이나 컴포넌트는 camelCase로 접근하여 모두 undefined 렌더링.
해결
notifications.jsx에 헬퍼 함수 추가. NotificationContext는 수정하지 않음.
Bug 03 icon 컬럼이 null이어서 아이콘 미표시
원인
DB에 icon 컬럼값이 없음.
해결
getIcon()에서 type별 기본 아이콘 fallback 처리.
Bug 04 Property 'setUserId' doesn't exist (Hermes 엔진 에러)
원인
useEffect dependency array에 setUserId 누락.
해결
[setUserId]를 dependency array에 명시적으로 추가.

향후 고려사항

  • icon / icon_color / icon_bg 컬럼을 DB에서 적절히 채워 넣으면 헬퍼의 기본값 fallback 없이 동작 가능.
  • 현재 헬퍼의 ICON_BY_TYPE에 type 종류가 추가될 경우 추후 별도 constants 파일로 분리 권장.
  • NotificationContext.jsx의 unreadCount는 is_read를 기준으로 계산하므로 현재 구조에서도 정상 동작.
  • 실시간 알림(addRealtimeNotification)으로 들어온 데이터도 헬퍼가 올바르게 처리함.
