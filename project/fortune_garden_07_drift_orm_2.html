01

작업 개요

SAD v1.1 및 DB ERD v1.1을 기반으로 Fortune Garden의 Drift ORM 데이터 계층(L3)을 v1.0에서 v1.1로 재작성하였습니다. CODEF API 방식 폐기 → CSV 파일 가져오기 방식 전환이 핵심 변경사항이며, DB 스키마와 쿼리 구조가 함께 변경되었습니다.

작업 유형 Drift ORM 스키마 코드 v1.0 → v1.1 재작성
변경 동인 CODEF API 폐기 → CSV 파일 가져오기 방식 전환 (SAD v1.1)
테이블 변화 8개 → 9개  (+csv_parser_profiles, sync_history → import_history)
쿼리 변화 4개 → 6개  (+getParserProfile, +getImportHistory)
인덱스 변화 8개 → 9개  (idx_sync_profile_start 제거, 신규 2개 추가)
Seed Data app_settings 8개 → 7개  (auto_sync_enabled 제거)
의존성 변화 shelf · dio 제거  /  csv · file_picker 추가

03

핵심 변경 상세

3.1 테이블 변경 (tables.dart)

3.1.1 profiles — connectedIdEnc 컬럼 제거

CODEF API 인증 방식 폐기에 따라 Connected ID를 저장하는 컬럼을 제거하였습니다.

v1.0 BlobColumn connectedIdEnc — AES-256-GCM 암호화된 CODEF Connected ID
v1.1 해당 컬럼 삭제. profiles는 id, name, colorHex, defaultViewMode, createdAt 5개 컬럼만 유지.

3.1.2 sync_history 테이블 제거

자동 동기화 개념이 폐기됨에 따라 동기화 이력 테이블을 완전히 제거하였습니다.

제거된 컬럼: startedAt, completedAt, status (running|success|partial|failed), txnCount, errorMsg

3.1.3 csv_parser_profiles — 신규 추가 NEW

기관별 CSV 파싱 설정을 DB로 관리. 앱 업데이트 없이 신규 기관 CSV 형식을 추가·수정할 수 있습니다.

3.2 신규 쿼리 (app_database.dart)

5.5 getParserProfile() NEW

institution_code 기준으로 활성화된 파서 프로필을 반환합니다. SCR-008 CSV 가져오기 화면에서 계좌 선택 시 자동으로 호출합니다. 

Future<CsvParserProfile?> getParserProfile({
  required String institutionCode,
}) {
  return (select(csvParserProfiles)
        ..where((t) =>
            t.institutionCode.equals(institutionCode) &
            t.isActive.equals(1))
        ..limit(1))
      .getSingleOrNull();
}

5.6 getImportHistory() NEW

accountId 기준 최신 이력을 20건 반환합니다. SCR-007 계좌 관리 화면 ImportHistoryList에서 사용합니다. 

Future<List<ImportHistoryEntry>> getImportHistory({
  required int accountId,
  int limit = 20,
}) {
  return (select(importHistory)
        ..where((t) => t.accountId.equals(accountId))
        ..orderBy([(t) => OrderingTerm.desc(t.importedAt)])
        ..limit(limit))
      .get();
}

3.3 마이그레이션 전략 (onUpgrade)

schemaVersion을 1 → 2로 상향하고 v1 → v2 업그레이드 경로를 구현하였습니다. 

onUpgrade: (m, from, to) async {
  if (from < 2) {
    // 1. sync_history 테이블 제거
    await m.database.customStatement(
      'DROP TABLE IF EXISTS sync_history',
    );

    // 2. 신규 테이블 생성
    await m.createTable(csvParserProfiles);
    await m.createTable(importHistory);

    // 3. 신규 인덱스 생성
    await m.database.customStatement(
      'CREATE INDEX IF NOT EXISTS idx_import_account_date '
      'ON import_history (account_id, imported_at DESC)',
    );
    await m.database.customStatement(
      'CREATE INDEX IF NOT EXISTS idx_parser_institution '
      'ON csv_parser_profiles (institution_code)',
    );
  }
},

주의: profiles.connected_id_enc 컬럼 제거는 onUpgrade에서 처리하지 않습니다. SQLite 컬럼 삭제 지원(v3.35.0+)이 모든 기기에서 보장되지 않으므로 신규 설치(onCreate)에서만 컬럼 없이 생성되며, 기존 사용자 DB는 컬럼을 그대로 유지합니다(무해함).

04

발생한 오류 및 해결 내역

코드 작성 및 프로젝트 적용 과정에서 발생한 오류와 예방 수정 내역을 기록했습니다.

ERROR Target of URI hasn't been generated: 'app_database.g.dart'
원인 part 'app_database.g.dart' 선언은 있으나 build_runner를 실행하지 않아 파일이 생성되지 않은 상태. Dart 분석기가 코드 생성 파일 없음을 감지해 48개 이상의 연쇄 오류 발생.
해결 pubspec.yaml에 drift_dev·build_runner dev 의존성 추가 후 dart run build_runner build --delete-conflicting-outputs 실행.
ERROR Undefined name 'OrderingTerm'
원인 database_provider.dart에 import 'package:drift/drift.dart'가 누락됨. OrderingTerm, Category, Institution 등 drift 타입을 사용하지만 import 없이 작성되었음.
해결 database_provider.dart 상단에 import 'package:drift/drift.dart'; 한 줄 추가.
FIX Transactions.uniqueKeys 중복 선언 제거 (예방)
내용 txnHash 컬럼에 .unique()와 uniqueKeys 오버라이드가 동시 선언되어 build_runner 충돌 가능성 존재. uniqueKeys 블록을 제거하고 .unique() 하나만 유지.
FIX onUpgrade의 잘못된 addColumn 호출 제거 (예방)
내용 onUpgrade에서 이미 존재하는 컬럼(profiles.defaultViewMode)에 m.addColumn()을 호출하는 불필요한 코드 제거. 해당 컬럼은 v1.0부터 존재하므로 마이그레이션 불필요.

05

build_runner 실행 가이드

app_database.g.dart 생성을 위한 전체 실행 절차입니다. VS Code 터미널(Ctrl + `)에서 프로젝트 루트 위치에서 실행합니다.

Step 1 — 의존성 추가 (최초 1회)

# runtime 의존성
$ flutter pub add drift sqlite3_flutter_libs path_provider path \
    flutter_riverpod csv file_picker

# dev 의존성
$ flutter pub add --dev drift_dev build_runner

Step 2 — 코드 생성 (테이블 변경 시마다)

# 1회 빌드 (권장)
$ dart run build_runner build --delete-conflicting-outputs

# 파일 변경 감지 자동 재생성 (개발 중)
$ dart run build_runner watch --delete-conflicting-outputs

재실행 필요 시점: 테이블 클래스 수정, 쿼리 메서드 시그니처 변경, @DriftDatabase tables 목록 변경 시 반드시 재실행.

Step 3 — 생성 결과 확인

빌드 성공 시 app_database.g.dart가 자동 생성되며 아래 항목이 포함됩니다.

  • _$AppDatabase mixin — 테이블 접근자 및 DatabaseInfo
  • 9개 Companion 클래스 (TransactionsCompanion, CsvParserProfilesCompanion 등)
  • $ProfilesTable, $AccountsTable 등 런타임 테이블 클래스
