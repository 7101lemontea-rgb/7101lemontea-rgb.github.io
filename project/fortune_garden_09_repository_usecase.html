01

작업 개요

SAD v1.1 §4.3의 Repository 패턴 명세와 §4.2의 UseCase 정의를 기반으로 Application Layer와 Data Layer의 전체 구현 코드를 작성하였다. 오늘 작업으로 Data Layer(DB) 위에 비즈니스 로직 레이어가 완성되어 Presentation Layer(화면) 구현이 가능한 상태가 되었다.

항목 내용
작업 순서 1단계 Repository → 2단계 UseCase (SAD §4.2 우선순위 순)
Repository 인터페이스 7개 domain/ + 구현체 7개 data/ + Provider 1개
UseCase 7개 — ProfileUseCase, ViewModeUseCase, CategoryClassifyUseCase, CsvParserUseCase, CsvImportUseCase, TransactionUseCase, ReportUseCase
추가 엔티티 ViewMode enum, TransactionFilter, ParsedTransaction, ImportResult
신규 패키지 crypto (SHA-256), intl (DateFormat)

03

Repository 레이어

UseCase가 DB에 직접 의존하지 않도록 추상 인터페이스로 감싸는 레이어. 테스트 시 Mock으로 교체 가능.

3.1 인터페이스 (domain/repositories/)

인터페이스 주요 메서드 비고
IProfileRepository getAll, getById, upsert, delete 최대 2개 프로필
IAccountRepository getByProfile, getById, upsert, updateBalance, delete 계좌번호 AES-256 암호화
ITransactionRepository getList(filter), getById, upsert, delete, existsByHash TransactionFilter 파라미터
ICategoryRepository getAll, getById, upsert, delete, getRules, upsertRule, deleteRule 자기참조 FK 수동 처리
ICsvParserProfileRepository getByInstitution, getAll, upsert, deactivate 삭제 대신 비활성화
IImportHistoryRepository insert, getByAccount limit 파라미터 지원
ISettingsRepository get(defaultValue), set, delete Key-Value 저장소

3.2 추가 모델 — TransactionFilter

ITransactionRepository.getList()의 조건 파라미터. 모든 필드 optional — null이면 해당 조건 미적용. 

class  TransactionFilter {
  final int?  profileId;   // null = 전체 프로필 (통합 뷰)
  final int?  accountId;   // 특정 계좌만
  final int?  categoryId;  // 특정 카테고리만
  final int?  fromDate;    // 시작일 Unix ms (이상)
  final int?  toDate;      // 종료일 Unix ms (미만)
  final bool? isManualOnly; // true=수동입력만, false=CSV만, null=전체
  final int   offset;     // 페이지네이션 (기본 0)
  final int   limit;      // 페이지네이션 (기본 50)
}

3.3 구현체 (data/repositories/) 주요 특이사항

category_repository.dart

delete()를 DB 트랜잭션으로 처리. 먼저 하위 카테고리의 parentId를 NULL로 초기화 후 삭제. (Drift 자기참조 FK 미지원)

transaction_repository.dart

upsert()는 app_database.insertTransactionIfNotExists()를 재사용. txnHash 충돌 시 자동 무시.

csv_parser_profile_repository.dart

getByInstitution()은 app_database.getParserProfile() 쿼리 메서드를 위임 호출.

import_history_repository.dart

getByAccount()는 app_database.getImportHistory() 쿼리 메서드를 위임 호출.

repository_providers.dart

7개 Repository를 Riverpod Provider로 노출. UseCase는 이 Provider를 통해 인스턴스 획득.

04

UseCase 레이어

순수 Dart 코드. Flutter, SQLite, HTTP에 의존하지 않아 단위 테스트 용이. SAD v1.1 §4.2 기준.

4.1 ProfileUseCase

활성 프로필 전환 및 프로필 CRUD. app_settings의 active_profile_id 키로 영속 저장.

메서드 설명
getAll() 모든 프로필 목록 반환
getActiveProfile() app_settings active_profile_id 기준 현재 활성 프로필 반환
switchProfile(id) active_profile_id 설정값 업데이트
upsert(...) 프로필 생성 또는 수정. createdAt 자동 설정
delete(id) 프로필 삭제. CASCADE로 accounts·transactions 삭제

4.2 ViewModeUseCase

개인 뷰 / 통합 뷰 전환. app_settings의 default_view_mode 키로 영속 저장. 

// ViewMode enum (domain/entities/view_mode.dart)
enum ViewMode {
  personal,  // WHERE profile_id = :activeProfileId
  combined;  // profile_id 조건 없음 (두 프로필 합산)

  static ViewMode fromString(String value) => ...
  String get value => name; // DB 저장용 문자열
}

4.3 CategoryClassifyUseCase

거래처명 LIKE 검색으로 카테고리 자동 분류. DB ERD v1.1 §6.3 쿼리 패턴 기준.

정렬 우선순위: ① 개인 규칙(profileId != null) 우선  ② priority 내림차순 

Future<int?> classify({
  required String merchant,
  required int    profileId,
}) async {
  final rules = await _categoryRepo.getRules(profileId: profileId);
  // 개인 규칙 우선 + priority 내림차순 정렬
  for (final rule in sorted) {
    if (merchant.contains(rule.keyword)) return rule.categoryId;
  }
  return null; // 미분류
}

4.4 CsvParserUseCase

파서 프로필에 따라 CSV 파일을 읽어 ParsedTransaction 목록으로 변환. Isolate에서 실행되는 순수 Dart 코드.

①
파일 읽기

file.readAsBytes()로 바이트 읽기

②
인코딩 변환

UTF-8: dart:convert / EUC-KR: charset_converter (TODO)

③
CSV 파싱

csv 패키지 CsvToListConverter

④
헤더 매핑

parserProfile.dateCol·amountCol·merchantCol로 컬럼 인덱스 탐색

⑤
행 변환

intl DateFormat으로 날짜 파싱, 금액 쉼표·공백 제거 후 int 변환

⑥
오류 처리

파싱 실패 행은 continue (errorRows 집계는 CsvImportUseCase)

EUC-KR 디코딩: charset_converter 패키지 도입 후 _decode() 메서드 내 TODO 주석 구간을 완성. 현재는 UTF-8 fallback으로 동작.

4.5 CsvImportUseCase

CSV 가져오기 전체 오케스트레이션. SAD v1.1 §5.2 흐름 전체 구현. 

Future<ImportResult> execute({
  required File   file,
  required int    profileId,
  required int    accountId,
  required String institutionCode,
}) async {
  // 1. 파서 프로필 자동 매칭 (institution_code 기준)
  // 2. Dart Isolate에서 CsvParserUseCase.parse() 실행
  // 3. 각 거래: SHA-256 해시 계산 → 중복 체크 → 카테고리 분류 → INSERT
  // 4. import_history 기록 (신규/중복/오류 건수)
  return ImportResult(...);
}
단계 파서 프로필 매칭

institutionCode → ICsvParserProfileRepository.getByInstitution()

단계 Isolate 실행

Isolate.run()으로 CsvParserUseCase.parse() 호출. UI 스레드 블로킹 방지

단계 txn_hash 계산

SHA-256(accountId|txnDate|amount|merchant). crypto 패키지 사용

단계 중복 체크

ITransactionRepository.existsByHash() — true면 duplicateRows++, skip

단계 카테고리 분류

CategoryClassifyUseCase.classify() — null이면 미분류

단계 import_history

신규/중복/오류 건수 집계 후 IImportHistoryRepository.insert()

4.6 TransactionUseCase

거래 내역 CRUD. 수동 입력(is_manual=1)과 메타 수정(카테고리·메모) 지원.

메서드 설명
getList(filter) TransactionFilter 조건으로 거래 목록 반환 (페이지네이션 포함)
getById(id) 단일 거래 조회
upsert(...) 수동 거래 생성 또는 수정. hash를 manual_ 접두사로 생성
delete(id) 거래 삭제
updateMeta(...) 카테고리 또는 메모만 수정. 거래 상세 화면(SCR-004)에서 사용

4.7 ReportUseCase

월별 수입/지출 요약. ViewMode에 따라 개인 뷰(단일 프로필) 또는 통합 뷰(합산) 결과 반환.

반환 모델 MonthlySummary 필드 설명
profileId 프로필 고유 ID
profileName 프로필 표시 이름
colorHex 프로필 색상 HEX (통합 뷰 태그 표시용)
income 수입 합계 (양수)
expense 지출 합계 (음수 raw값)
net income + expense (순수지)

ReportUseCase는 app_database.getMonthlySummary()·getCombinedDashboard() 쿼리를 직접 재사용. 별도 SQL 없음.
