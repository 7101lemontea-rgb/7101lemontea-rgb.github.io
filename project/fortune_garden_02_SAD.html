01

문서 개요

1.1 목적

본 문서는 Flutter 기반 2인 가구용 자동 동기화 가계부 애플리케이션 Fortune Garden의 전체 시스템 아키텍처를 정의한다. 개발팀(바이브코딩 포함)이 일관된 구조로 구현할 수 있도록 컴포넌트 구성, 계층 간 인터페이스, 데이터 흐름, 보안 모델, 배포 구조를 명세한다.

1.2 범위

  • 대상 플랫폼: Windows 10/11 (Flutter Desktop), Android 8.0 이상 (Flutter Mobile)
  • 핵심 기능 범위: CODEF API 연동, 2인 프로필 관리, 자동 동기화, 거래 내역 조회, 지출 리포트
  • 제외 범위: 클라우드 서버, 웹 인터페이스, 3인 이상 멀티유저

1.3 참조 문서

문서명 버전 비고
요구사항 정의서 Fortune Garden v1.1 2026-03-07 수정안

02

아키텍처 원칙 및 목표

2.1 핵심 설계 원칙

원칙 설명
Privacy-First (로컬 우선) 모든 금융 데이터는 사용자 PC의 SQLite에만 저장하며, 외부 서버로 어떠한 금융 정보도 전송하지 않는다.
단방향 의존성 Presentation → Domain → Data → External 방향으로만 의존한다. 역방향 참조는 금지한다.
UI Non-blocking 네트워크 I/O 및 DB 쓰기 작업은 모두 Dart Isolate 또는 비동기(async/await)로 처리하여 UI 스레드를 차단하지 않는다.
Secret Key 은닉 CODEF Client Secret은 Flutter 번들에 포함하지 않고 로컬 Shelf 서버 내부에 격리하여, 앱 번들 분석으로 키가 노출되지 않도록 한다.
2인 프로필 대칭 설계 모든 데이터 모델(accounts, transactions, sync_history)은 profile_id를 외래 키로 가지며, 프로필 간 데이터는 완전히 격리된다.
확장 가능한 기관 코드 관리 금융기관 코드는 DB 테이블로 관리하여 코드 수정 없이 기관을 추가·삭제할 수 있다.

2.2 아키텍처 목표 품질 속성

품질 속성 목표 주요 아키텍처 결정
성능 대시보드 초기 로드 2초 이내, 동기화 중 60fps 유지 Isolate 분리, SQLite 인덱스, ListView.builder 지연 로딩
보안 AES-256 암호화, Secret Key 은닉 OS Keychain 통합, Shelf 프록시 구조
가용성 오프라인 모드에서도 조회·수정 가능 로컬 SQLite 캐시, 네트워크 상태 감지 레이어
유지보수성 Clean Architecture 레이어 분리, 테스트 커버리지 70%+ Repository 패턴, UseCase 분리, 의존성 주입(Riverpod)
확장성 기관 코드·카테고리 규칙 DB 관리 기관 코드 테이블, category_rules 테이블

03

전체 시스템 구조

3.1 계층형 아키텍처 개요

Fortune Garden은 5개의 논리 계층으로 구성된다. 각 계층은 인접 계층과만 통신하며, 계층을 건너뛰는 직접 호출은 허용하지 않는다.

L1
PresentationFlutter UI

화면 렌더링, 사용자 입력 처리, 상태 구독(Riverpod). 비즈니스 로직 포함 금지.

L2
ApplicationUseCase / Service

비즈니스 유스케이스 실행. 동기화 오케스트레이션, 카테고리 분류, 프로필 전환 로직.

L3
InfrastructureShelf Local Server

CODEF API 프록시, Secret Key 은닉, OAuth 토큰 관리. localhost:8765에서만 수신.

L4
DataRepository + SQLite

데이터 영속성. drift ORM 기반 SQLite 접근, Repository 패턴으로 추상화.

L5
ExternalCODEF API

금융기관 스크래핑 및 인증 중계. REST API / HTTPS. 앱 외부 서비스.

04

계층별 상세 설계

4.1 Presentation Layer (L1) — Flutter UI

4.1.1 상태 관리

상태 관리 라이브러리로 Riverpod 2.x를 사용한다. 각 화면은 ConsumerWidget 또는 ConsumerStatefulWidget으로 구현하며, Provider를 통해 UseCase 결과를 구독한다.

Provider 이름 상태 타입 역할
activeProfileProvider Profile? 현재 선택된 프로필 (개인 뷰용)
viewModeProvider ViewMode (personal/combined) 개인 뷰 / 통합 뷰 전환 상태
syncStateProvider SyncState (idle/syncing/done/error) 동기화 진행 상태 및 결과
transactionListProvider AsyncValue<List<Transaction>> 거래 내역 목록 (필터 파라미터 포함)
dashboardSummaryProvider DashboardSummary 총 잔액, 이번달 수입/지출 요약

4.1.2 화면 구성 및 네비게이션

Windows Desktop은 NavigationRail 기반 사이드바 레이아웃을, Android는 BottomNavigationBar 레이아웃을 사용한다. 공통 라우팅은 go_router 패키지로 관리한다.

4.2 Application Layer (L2) — UseCase / Service

Domain Layer에 해당하며, 순수 Dart 코드로만 구성한다. Flutter, SQLite, HTTP 패키지에 의존하지 않아 단위 테스트가 용이하다.

UseCase 클래스 메서드 설명
SyncUseCase execute(profileId) 단일 프로필의 전체 기관 동기화 오케스트레이션. Isolate에서 실행.
SyncAllProfilesUseCase execute() 두 프로필을 순차 동기화. SQLite WAL 모드로 잠금 충돌 방지.
CategoryClassifyUseCase classify(merchant) 거래처명으로 category_rules 테이블 조회 후 카테고리 반환.
AuthUseCase initiateAuth(profileId, method) CODEF 인증 시작. Shelf 서버로 인증 요청 위임.
ProfileUseCase switchProfile(id) / getActiveProfile() 활성 프로필 전환, 프로필 CRUD.
ViewModeUseCase setMode(ViewMode) / getMode() 개인 뷰 / 통합 뷰 전환 상태 관리.
TransactionUseCase getList(filter) / upsert() / delete() 거래 내역 CRUD 및 필터 쿼리.
ReportUseCase getMonthlySummary(month, viewMode) 뷰 모드에 따라 단일 또는 합산 리포트 데이터 반환.

4.3 Infrastructure Layer (L3) — Shelf Local Server

4.3.1 역할 및 구동 방식

Shelf 서버는 Flutter 앱 시작 시 별도 Dart Isolate에서 localhost:8765 포트로 구동된다. Flutter UI는 HTTP 클라이언트(dio 패키지)를 통해 Shelf 서버와만 통신하며, CODEF API를 직접 호출하지 않는다. 이 구조로 Client Secret Key가 Flutter 번들에 포함되지 않는다.

4.3.2 API 엔드포인트 명세

Method 경로 파라미터 설명
POST /auth/initiate { profileId, method } CODEF 간편인증 시작. 인증 URL 또는 QR 코드 반환.
POST /auth/callback { code, state } CODEF 인증 콜백 처리. Connected ID 발급 및 암호화 저장.
POST /sync { profileId, fromDate } 지정 프로필의 전체 기관 동기화 트리거. SSE로 진행 상태 스트리밍.
GET /sync/status { jobId } 진행 중인 동기화 작업 상태 조회.
GET /health - 서버 구동 상태 확인. 앱 시작 시 준비 완료 체크에 사용.

4.3.3 Secret Key 관리 방식

CODEF Client ID와 Secret Key는 아래 우선순위로 로드된다. 어느 방법을 사용하든 Shelf 서버 프로세스 메모리 내에만 존재하며, 외부로 노출되지 않는다.

  • OS Keychain (Windows Credential Manager / Android Keystore) — 최우선
  • 앱 데이터 디렉터리의 AES-256 암호화 설정 파일 (codef_config.enc)
  • 환경 변수 (CODEF_CLIENT_ID, CODEF_SECRET_KEY) — 개발 환경 전용

4.4 Data Layer (L4) — Repository + SQLite

4.4.1 Repository 패턴

모든 데이터 접근은 추상 Repository 인터페이스를 통해 이루어진다. UseCase는 인터페이스에만 의존하므로 테스트 시 Mock Repository로 교체 가능하다.

Repository 인터페이스 주요 메서드

IAccountRepository getByProfile(profileId), upsert(account), updateBalance(id, balance)
ITransactionRepository getList(filter), upsert(txn), delete(id), existsByHash(hash)
ICategoryRepository getAll(), upsert(cat), delete(id), getRules()
ISyncHistoryRepository insert(record), getLatestByProfile(profileId)
ISettingsRepository get(key), set(key, value), delete(key)

4.4.2 SQLite 데이터베이스 상세 스키마

drift ORM을 사용하며, DB 파일명은 fortune_garden.db이다. 저장 경로는 OS별 표준 앱 데이터 디렉터리를 따른다.

컬럼명 타입 제약 설명

기타 테이블

테이블명 주요 컬럼 설명

05

주요 데이터 흐름

5.1 앱 시작 시퀀스

단계 주체 동작
1 Flutter App 앱 프로세스 시작. main() 진입.
2 Shelf Server 별도 Isolate에서 Shelf 서버 구동 (localhost:8765). GET /health 응답 확인.
3 Data Layer SQLite DB 열기. 미적용 마이그레이션 자동 실행.
4 Application profiles 테이블 조회. 프로필 0개 → 온보딩 화면. 1개 이상 → 대시보드.
5 Application FR-CFG-06에 따라 기본 뷰 모드(개인/통합) 로드 후 viewModeProvider 초기화.
6 SyncUseCase FR-SYNC-01: 자동 동기화 활성화 설정 확인 후 SyncAllProfilesUseCase 실행(Isolate).
7 Presentation syncStateProvider 구독 → 대시보드에 진행 상태 실시간 표시.

5.2 2인 프로필 동기화 흐름

두 프로필의 동기화는 순차(Sequential) 실행을 원칙으로 한다. SQLite의 단일 쓰기 잠금 특성상 병렬 실행 시 잠금 오류가 발생할 수 있기 때문이다. WAL(Write-Ahead Logging) 모드를 활성화하여 읽기와 쓰기의 동시성을 최대화한다.

  • SyncAllProfilesUseCase.execute() 호출
  • Profile A의 SyncUseCase.execute(profileIdA) 실행
  • Shelf 서버로 POST /sync { profileId: A } 요청
  • Shelf 서버가 Profile A의 Connected ID 복호화 후 CODEF API 호출
  • 기관별 거래 내역 수신 → txn_hash 중복 체크 → SQLite BEGIN TRANSACTION → 일괄 INSERT → COMMIT
  • sync_history 기록 (Profile A)
  • Profile A 완료 후 Profile B의 SyncUseCase.execute(profileIdB) 실행 (동일 흐름)
  • 두 프로필 완료 후 dashboardSummaryProvider 갱신 → UI 리렌더링

5.3 개인 뷰 / 통합 뷰 전환 흐름

뷰 모드 전환은 viewModeProvider 값 변경 → transactionListProvider / dashboardSummaryProvider 자동 재계산의 Riverpod 반응형 흐름으로 처리된다. DB 재쿼리가 아닌 파라미터 변경만으로 즉시 반영된다.

뷰 모드 데이터 쿼리 방식 표시 내용
개인 뷰 (Personal) WHERE profile_id = :activeProfileId 활성 프로필의 계좌·거래만 표시. 프로필 색상 태그 없음.
통합 뷰 (Combined) profile_id IN (1, 2) — 조건 없음 두 프로필 합산. 각 거래 항목에 프로필 색상 태그 표시.

06

보안 아키텍처

6.1 암호화 계층

대상 데이터 암호화 방식 키 저장 위치
Connected ID (profiles.connected_id_enc) AES-256-GCM OS Keychain (Windows Credential Manager / Android Keystore)
계좌번호 (accounts.account_number_enc) AES-256-GCM OS Keychain (동일 마스터 키 사용)
CODEF Secret Key 메모리 내 보관 (암호화 저장 시 AES-256) OS Keychain 또는 암호화 설정 파일
SQLite DB 파일 전체 SQLCipher (선택적 적용) OS Keychain 마스터 키

6.2 네트워크 보안

  • Shelf 서버는 127.0.0.1:8765에만 바인딩. 외부 네트워크 인터페이스 바인딩 금지 (FR-SEC-03, NFR-SEC-02).
  • CODEF API 호출은 모두 HTTPS. 인증서 검증 활성화.
  • Flutter UI ↔ Shelf 서버 통신은 localhost 루프백이므로 암호화 불필요. 단, 요청 헤더에 앱 내부 토큰(X-App-Token) 포함하여 타 프로세스의 무단 호출 방지.

6.3 PIN 잠금 흐름

  • 앱 시작 시 FR-SEC-04 PIN 설정 여부 확인.
  • PIN 활성화 상태이면 PinLockScreen 표시. 정확한 PIN 입력 시 앱 진입.
  • 앱이 백그라운드로 전환 후 30초 경과 시 자동 잠금 (설정 가능).
  • PIN은 SHA-256 해시로 저장. 평문 PIN은 메모리에서 즉시 해제.

아키텍처 결정 기록 (ADR)

주요 기술 결정의 배경과 대안을 기록한다.

ADR-01Shelf 서버를 별도 Isolate로 구동

고려한 대안

Flutter 앱 메인 스레드에서 구동

채택 이유

메인 스레드 차단 방지. 서버 오류가 UI에 영향을 주지 않음.

ADR-02drift ORM 채택

고려한 대안

sqflite 직접 사용, Hive

채택 이유

타입 안전 쿼리, 자동 마이그레이션, Isolate 지원이 가계부처럼 쿼리가 많은 앱에 적합.

ADR-032인 동기화를 순차 실행

고려한 대안

두 프로필 병렬 동기화

채택 이유

SQLite 단일 쓰기 잠금 회피. 총 소요 시간 증가는 미미 (각 프로필 5~7초 × 2).

ADR-04Riverpod 상태 관리 채택

고려한 대안

BLoC, Provider, GetX

채택 이유

UseCase 계층과의 결합도가 낮고, 테스트 시 Provider override가 용이.

ADR-05금액 저장 단위: 정수(원)

고려한 대안

실수(double)

채택 이유

부동소수점 오차 방지. 금융 데이터는 반드시 정수로 저장.

ADR-06Connected ID를 DB에 암호화 저장

고려한 대안

OS Keychain에만 저장

채택 이유

여러 프로필의 Connected ID를 구조적으로 관리하기 위해 DB 저장 + 암호화 키만 Keychain에 보관.
