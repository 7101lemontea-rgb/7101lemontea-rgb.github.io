01

문서 개요

1.1 목적

본 문서는 Flutter 기반 2인 가구용 가계부 애플리케이션 Fortune Garden의 전체 시스템 아키텍처를 정의한다. 개발팀이 일관된 구조로 구현할 수 있도록 컴포넌트 구성, 계층 간 인터페이스, 데이터 흐름, 보안 모델, 배포 구조를 명세한다.

1.2 범위

  • 대상 플랫폼: Windows 10/11 (Flutter Desktop), Android 8.0 이상 (Flutter Mobile)
  • 핵심 기능 범위: CSV 파일 가져오기, 2인 프로필 관리, 거래 내역 조회, 카테고리 분류, 지출 리포트
  • 제외 범위: 클라우드 서버, 웹 인터페이스, 3인 이상 멀티유저, 외부 API 연동

02

아키텍처 원칙 및 목표

2.1 핵심 설계 원칙

원칙설명
Privacy-First (로컬 우선)모든 금융 데이터는 사용자 PC의 SQLite에만 저장하며, 외부 서버로 어떠한 금융 정보도 전송하지 않는다.
단방향 의존성Presentation → Domain → Data → External 방향으로만 의존한다. 역방향 참조는 금지한다.
UI Non-blockingCSV 파싱 및 DB 쓰기 작업은 모두 Dart Isolate 또는 비동기(async/await)로 처리하여 UI 스레드를 차단하지 않는다.
2인 프로필 대칭 설계모든 데이터 모델(accounts, transactions, import_history)은 profile_id를 외래 키로 가지며, 프로필 간 데이터는 완전히 격리된다.
확장 가능한 파서 프로필 관리CSV 파서 프로필은 DB 테이블(csv_parser_profiles)로 관리하여 코드 수정 없이 신규 기관 추가가 가능하다.
완전 로컬 독립 동작외부 API 의존성을 제거하여 네트워크 없이도 앱의 모든 핵심 기능이 동작한다.

03

전체 시스템 구조

3.1 계층형 아키텍처 개요

Fortune Garden은 v1.2에서 4개의 논리 계층으로 구성된다. v1.1의 Infrastructure(Shelf Local Server) 계층(L3)이 제거되었다. 각 계층은 인접 계층과만 통신하며, 계층을 건너뛰는 직접 호출은 허용하지 않는다.

L1
PresentationFlutter UI

화면 렌더링, 사용자 입력 처리, 상태 구독(Riverpod). 비즈니스 로직 포함 금지.

L2
ApplicationUseCase / Service

비즈니스 유스케이스 실행. CSV 파싱 오케스트레이션, 카테고리 분류, 프로필 전환 로직.

L3
DataRepository + SQLite

데이터 영속성. drift ORM 기반 SQLite 접근, Repository 패턴으로 추상화.

L4
External로컬 파일 시스템

사용자가 가져온 CSV/엑셀 파일. 네트워크 없이 동작. 외부 API 의존성 없음.

아래는 Fortune Garden v1.2의 주요 컴포넌트 간 관계를 나타낸다. Shelf Server 계층이 제거되고 CSV Parser가 Application 계층 내로 통합되었다.

Domain Layer에 해당하며, 순수 Dart 코드로만 구성한다. Flutter, SQLite, HTTP 패키지에 의존하지 않아 단위 테스트가 용이하다.

CsvParserUseCaseparse(file, parserProfile)파서 프로필에 따라 CSV 파일을 읽어 거래 목록으로 변환. 기관별 인코딩/컬럼 처리.

05

주요 데이터 흐름

5.1 앱 시작 시퀀스

단계주체동작
1Flutter App앱 프로세스 시작. main() 진입.
2Data LayerSQLite DB 열기. 미적용 마이그레이션 자동 실행.
3Applicationprofiles 테이블 조회. 프로필 0개 → 초기 설정 화면. 1개 이상 → 대시보드.
4ApplicationFR-CFG-05에 따라 기본 뷰 모드(개인/통합) 로드 후 viewModeProvider 초기화.
5Presentation대시보드 렌더링. 로컬 캐시 데이터 즉시 표시.

5.2 CSV 가져오기 흐름

  • 사용자가 금융기관에서 CSV 파일 내보내기 후 파일 선택(탐색기 열기 또는 드래그 앤 드롭)
  • 대상 프로필 및 계좌 선택. 파서 프로필 자동 매칭 (institution_code 기준)
  • CsvImportUseCase.execute() 호출 → Dart Isolate에서 파싱 시작
  • 각 행의 txn_hash(SHA-256) 계산 → DB 중복 체크(existsByHash)
  • 신규 거래만 SQLite BEGIN TRANSACTION → 일괄 INSERT → COMMIT
  • CategoryClassifyUseCase로 카테고리 자동 분류(category_rules LIKE 매칭)
  • import_history 기록 (신규/중복/오류 건수)
  • 완료 후 csvImportStateProvider 갱신 → Flutter UI 결과 표시 및 화면 갱신

5.3 수동 거래 입력 흐름

  • 사용자가 수동 입력 화면에서 거래 정보 입력 (날짜, 금액, 거래처, 카테고리 등)
  • is_manual=1 플래그와 함께 SQLite transactions 테이블에 직접 저장
  • 저장 완료 후 transactionListProvider 갱신 → 거래 내역 목록 즉시 반영

5.4 개인 뷰 / 통합 뷰 전환 흐름

뷰 모드 전환은 viewModeProvider 값 변경 → transactionListProvider / dashboardSummaryProvider 자동 재계산의 Riverpod 반응형 흐름으로 처리된다. DB 재쿼리가 아닌 파라미터 변경만으로 즉시 반영된다.

뷰 모드데이터 쿼리 방식표시 내용
개인 뷰 (Personal)WHERE profile_id = :activeProfileId활성 프로필의 계좌·거래만 표시. 프로필 색상 태그 없음.

보안 아키텍처

6.1 암호화 계층

대상 데이터암호화 방식키 저장 위치
계좌번호 (accounts.account_number_enc)AES-256-GCMOS Keychain (Windows Credential Manager / Android Keystore)
SQLite DB 파일 전체SQLCipher (선택적 적용)OS Keychain 마스터 키

6.2 로컬 보안

  • 모든 금융 데이터는 외부 서버로 전송되지 않으며 사용자 PC의 SQLite에만 보관된다.
  • 계좌번호 등 민감 정보는 AES-256-GCM으로 암호화하여 저장한다.
  • 암호화 키는 OS Keychain에만 저장하여 앱 외부에서 접근 불가하도록 한다.
  • 로그에 민감 정보(계좌번호, 금액 등)가 평문으로 출력되지 않도록 한다.

6.3 PIN 잠금 흐름

  • 앱 시작 시 FR-SEC-03 PIN 설정 여부 확인.
  • PIN 활성화 상태이면 PinLockScreen 표시. 정확한 PIN 입력 시 앱 진입.
  • 앱이 백그라운드로 전환 후 30초 경과 시 자동 잠금 (설정 가능).
  • PIN은 SHA-256 해시로 저장. 평문 PIN은 메모리에서 즉시 해제.

아키텍처 결정 기록 (ADR)

주요 기술 결정의 배경과 대안을 기록한다.

ADR-01CSV 파일 가져오기 방식 채택

고려한 대안

CODEF API 자동 동기화, 핀테크 오픈API

채택 이유

개인 개발자 환경의 이용 기간 제한·비용·사업자 요건 없이 지속 가능. 국내 대부분의 금융기관이 CSV 내보내기를 공식 지원.

ADR-02drift ORM 채택

고려한 대안

sqflite 직접 사용, Hive

채택 이유

타입 안전 쿼리, 자동 마이그레이션, Isolate 지원이 가계부처럼 쿼리가 많은 앱에 적합.

ADR-03CSV 파싱을 Dart Isolate에서 실행

고려한 대안

메인 스레드 동기 처리

채택 이유

UI 블로킹 방지. 대용량 파일(수천 건) 처리 시 60fps 유지 목표 달성.

ADR-04Riverpod 상태 관리 채택

고려한 대안

BLoC, Provider, GetX

채택 이유

UseCase 계층과의 결합도가 낮고, 테스트 시 Provider override가 용이.

ADR-05금액 저장 단위: 정수(원)

고려한 대안

실수(double)

채택 이유

부동소수점 오차 방지. 금융 데이터는 반드시 정수로 저장.

ADR-06파서 프로필을 DB 테이블로 관리

고려한 대안

코드에 하드코딩

채택 이유

앱 업데이트 없이 신규 기관 CSV 형식 대응 가능. category_rules와 동일한 패턴으로 일관성 유지.
