02. 네이버 지도 API 및 식당 검색

2.1. 시스템 아키텍처 (Proxy Structure)

보안 및 일관성을 위해 Flutter가 직접 API를 호출하지 않고 FastAPI를 경유하는 프록시 구조를 채택했습니다. 

// 요청 흐름
Flutter (geolocator) → GPS 획득
   ↓ HTTP GET (lat, lng)
FastAPI (Reverse Geocoding) → 행정동 추출
   ↓ asyncio.gather (병렬 키워드 검색)
Naver Local API → 식당 리스트 (최대 200개)

2.2. 설계 결정 사항

  • 엔진 교체: Web 미지원 flutter_naver_map 대신 flutter_map 도입.
  • 검색 일관성: 블로그 검색 키워드와의 정합성을 위해 카카오 API 대신 네이버 로컬 API 사용.
  • 좌표 변환: 네이버 특유의 카텍 좌표를 WGS84(위경도)로 변환하는 로직 구현.

03. 데이터 정제 및 ELECTRA 통합

3.1. HTML 태그 정제 (_clean)

검색 결과에 포함된 <b> 태그와 HTML 엔티티를 제거하여 사용자 가독성을 확보했습니다. 

def _clean(text: str) -> str:
    text = re.sub(r"<[^>]+>", "", text) # 태그 제거
    text = unescape(text)                # 엔티티 디코딩
    return text.strip()

3.2. ELECTRA 모델 백엔드 통합

GPT 기반 판별에서 파인튜닝된 koelectra-base-v3 모델을 병행 사용할 수 있도록 개편했습니다.

  • Lifespan: 서버 시작 시 모델을 메모리에 1회 로드하여 추론 속도 최적화.
  • Labeling: 0(비광고), 1(광고)로 분류하며 확률값(adProbability)을 함께 반환.
  • Dependency: numpy<2.0 다운그레이드로 torch와의 충돌 방지.
