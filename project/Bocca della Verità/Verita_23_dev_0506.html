Conversion Pipeline

02. 변환 파이프라인

전체 흐름

best_model_N.pt (학습된 가중치)
    ↓ BertForSequenceClassification.load_state_dict()

HF 모델 (메모리)
    ↓ model.save_pretrained()

model.safetensors (올바른 HF 형식)
    ↓ OVModelForSequenceClassification.from_pretrained(export=True)
    ↓ nncf.quantize(preset=MIXED)

OpenVINO INT8
(openvino_model.xml / .bin)

⚠️ 주의: save_file(state_dict) 방식으로 저장하면 키 구조 불일치로 모델이 항상 0만 예측하는 문제가 발생합니다. 반드시 model.save_pretrained() 방식으로 저장해야 합니다.

단계별 코드

Step 1. PT → Safetensors 변환

import torch
from safetensors.torch import save_file
import os

os.makedirs("./kcbert_finetuned", exist_ok=True)

state_dict = torch.load(
    "model.pt",
    map_location="cpu"
)

state_dict = {
    k: v.contiguous()
    for k, v in state_dict.items()
}

save_file(
    state_dict,
    "./kcbert_finetuned/model.safetensors"
)

print("PT → Safetensors 완료!")

Step 4. Safetensors → OpenVINO INT8 변환

from optimum.intel import (
    OVModelForSequenceClassification
)

from transformers import AutoTokenizer
from datasets import Dataset
import pandas as pd
import numpy as np
import nncf

from openvino import serialize
import os

tokenizer = AutoTokenizer.from_pretrained(
    "./kcbert_finetuned"
)

# 캘리브레이션 데이터 준비
df = pd.read_csv("data.csv")

df["text"] = (
    df["review_title"]
    + " "
    + df["review_description"]
)

df = (
    df[["text"]]
    .dropna()
    .reset_index(drop=True)
)

calib_dataset = Dataset.from_pandas(
    df,
    preserve_index=False
)

calib_dataset = calib_dataset.select(
    range(
        min(
            300,
            len(calib_dataset)
        )
    )
)

def preprocess_fn(examples):
    return tokenizer(
        examples["text"],
        max_length=128,
        padding="max_length",
        truncation=True
    )

processed_dataset = calib_dataset.map(
    preprocess_fn,
    batched=True
)

processed_dataset = (
    processed_dataset.remove_columns(
        ["text"]
    )
)

processed_dataset.set_format("numpy")

# 배치 차원 추가 필수
def collate_fn(examples):
    return {
        "input_ids":
            np.array(
                examples["input_ids"]
            ).reshape(1, -1),

        "attention_mask":
            np.array(
                examples["attention_mask"]
            ).reshape(1, -1),

        "token_type_ids":
            np.array(
                examples[
                    "token_type_ids"
                ]
            ).reshape(1, -1)
    }

nncf_dataset = nncf.Dataset(
    processed_dataset,
    collate_fn
)

model = (
    OVModelForSequenceClassification
    .from_pretrained(
        "./kcbert_finetuned",
        export=True
    )
)

quantized_model = nncf.quantize(
    model.model,
    nncf_dataset,
    preset=(
        nncf
        .QuantizationPreset
        .MIXED
    )
)

os.makedirs(
    "./kcbert_openvino_int8",
    exist_ok=True
)

serialize(
    quantized_model,
    "./kcbert_openvino_int8/model_int8.xml"
)

print("INT8 변환 완료!")

INT8 Quantization Analysis

04. INT8 양자화 효과 분석

개요

  • FP32(32비트 부동소수점) → INT8(8비트 정수) 로 모델 가중치 및 활성화 변환
  • NNCF (Neural Network Compression Framework) 사용
  • 캘리브레이션 데이터셋 300개 샘플로 양자화 수행 (재학습 불필요)

이론적 vs 실측 성능 향상

구분 내용
이론적 최대 약 4배 속도 향상 (OpenVINO 공식 문서)
실측 (Intel CPU 기준) 약 3.6배 (Xeon 8280, Core i7-1165G7)

추론 속도 차이 원인 분석

두 INT8 모델 간 약 1.5배 이상의 속도 차이가 발생한 복합적 원인:

1. 커널 선택 (Kernel Selection)

동일한 MatMul 연산이라도 실행 시 선택되는 커널에 따라 성능이 크게 달라집니다. 고속 모델은 CPU의 SIMD 명령어 (AVX2, AVX-512, VNNI)를 활용하는 oneDNN 최적화 커널을 사용하는 반면, 저속 모델은 일반 커널을 사용할 가능성이 있습니다.

2. 연산자 융합 (Operator Fusion)

고속 모델은 MatMul + Bias + Activation(GELU) 연산이 하나의 fused kernel로 결합되어 실행되지만, 저속 모델은 분리 실행되어 커널 호출 횟수와 중간 결과 메모리 접근이 증가합니다.

3. 메모리 대역폭 효율 (Memory Bandwidth Utilization)

BERT 계열 모델은 연산보다 메모리 접근 비용이 더 큰 memory-bound 특성을 가집니다. INT8 weight는 메모리 사용량을 75% 감소시켜 cache locality를 향상시킵니다.

4. 데이터 레이아웃 및 Weight Prepacking

고속 모델은 실행 전 weight를 CPU 친화적인 구조로 재배열 (prepacking)하여 cache hit rate를 높이고 메모리 접근 패턴을 최적화합니다.

5. 실행 그래프 최적화 (Execution Graph Optimization)

OpenVINO 변환 과정에서 operator fusion, constant folding, subgraph rewriting이 수행되어 동일한 노드 수라도 실제 실행 경로가 달라집니다.

6. Precision vs Execution Decoupling

일부 모델은 INT8 weight를 사용하면서도 FP32로 변환 후 연산(dequantized execution)을 수행합니다. 이는 메모리 효율성(INT8)과 연산 안정성(FP32)을 동시에 확보하는 구조입니다.

결론: 모델의 추론 속도는 단순한 정밀도(INT8/FP32)나 노드 수가 아니라, 커널 최적화, 메모리 접근 효율, 연산자 융합 구조에 의해 결정됩니다.

Limitation & Future Work

07. 평가 한계 및 향후 계획

7-1. 현재 평가의 한계

문제 설명
데이터 누수 (Data Leakage) 학습 데이터와 평가 데이터가 동일하여 성능 과대평가 가능성
클래스 불균형 광고 513 : 비광고 768
기존 평가와 비교 불일치 기존(균형 1,000개) vs 현재(실제 분포 1,281개) 직접 비교 어려움

7-2. 기존 성능평가 수치가 낮아진 이유

기존 성능평가 (Recall 0.9800) 와 현재 ROC 분석 (Recall 0.9084) 차이는 모델 성능 저하가 아니라 아래 세 가지 원인으로 발생합니다.

  1. 테스트셋 규모 차이 — 1,000개 → 1,281개
  2. 클래스 불균형 — 500:500 (균형) → 513:768 (실제 분포)
  3. Threshold 변경 — 0.5 고정 → Youden's J 기준 변경

7-3. P99 신뢰도

샘플 수 P99 해당 샘플 수 신뢰도
200개 2개 ❌ 매우 낮음
1,000개 10개 ✅ 최소 기준
10,000개 100개 ✅✅ 안정적

7-4. 향후 계획

  • ☐ 새 데이터 수집 후 미학습 데이터로 성능 재평가
  • ☐ Train / Validation / Test 분리 기준 명확화
  • ☐ 실제 운영 환경 기준 최종 Threshold 확정
