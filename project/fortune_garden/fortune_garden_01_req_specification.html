01

프로젝트 개요

1.1 기본 정보

프로젝트명Flutter 기반 2인 가구용 가계부 Fortune Garden (포춘 가든)
대상 플랫폼Windows 10/11 (Flutter Desktop) / Android 8.0 이상 (Flutter Mobile)
핵심 기술Flutter, Dart, SQLite (drift ORM), CSV 파서, Riverpod, Dart Isolate
개발 유형2인 가구용 로컬 전용 가계부 애플리케이션
데이터 수집 방식CSV 파일 가져오기 + 수동 직접 입력 (외부 API 미사용)

1.2 프로젝트 목표

Fortune Garden 프로젝트는 여러 금융기관(시중 은행, 카드사 등)의 거래 내역을 CSV 파일 가져오기 및 수동 입력 방식으로 수집하고, 사용자의 로컬 환경에서 외부 서버 전송 없이 안전하게 자산을 관리할 수 있는 가계부 애플리케이션을 개발하는 것을 목표로 하고 있습니다.

  • 사용자가 보유한 여러 금융기관 계좌를 단일 인터페이스에서 통합 조회
  • 2인 각자의 계좌를 개별 또는 합산하여 조회 (개인 뷰 / 통합 뷰)
  • 각 금융기관 앱·웹에서 내보낸 CSV 파일을 가져와 거래 내역 자동 파싱 및 저장
  • 현금 거래 등 CSV에 포함되지 않는 항목은 수동 입력으로 보완
  • 모든 데이터는 로컬 SQLite에만 저장하여 개인 금융 정보 외부 유출 차단
  • 거래 내역의 카테고리 자동 분류 및 지출 리포트 제공으로 재정 관리 지원

1.3 배경 및 필요성

현재 시장에 출시된 가계부 애플리케이션 대부분은 클라우드 기반으로 동작하여 금융 데이터가 외부 서버에 저장되거나 전송됩니다. 개인 금융 정보의 민감성을 고려할 때, 데이터가 로컬에만 보관되는 완전한 프라이버시 중심의 가계부 애플리케이션이 필요합니다.

기존 자동 동기화 방식(CODEF API 등 금융 스크래핑 서비스)은 개인 개발자 환경에서 이용 기간 제한(데모 3개월), 사업자 등록 요건, 비용 발생 등의 현실적 한계가 있습니다. 이에 본 프로젝트는 국내 대부분의 금융기관이 공식 지원하는 거래내역 CSV 내보내기 기능을 활용하여 별도 API 계약 없이도 안정적으로 거래 내역을 수집하는 방식을 채택했습니다.

1.5 용어 정의

CSV 가져오기

각 금융기관의 앱 또는 웹사이트에서 제공하는 거래내역 CSV/엑셀 파일을 앱으로 불러오는 기능. 별도 API 계약 없이 사용 가능하며 대부분의 국내 금융기관이 지원한다.

CSV 파서

가져온 CSV 파일을 읽어 거래내역 데이터로 변환하는 모듈. 기관별 파일 포맷 차이(컬럼명, 날짜 형식, 인코딩 등)를 처리하는 기관별 파서 프로필을 내장한다.

파서 프로필

금융기관별 CSV 컬럼 구조, 날짜 형식, 인코딩, 구분자 등의 파싱 규칙을 정의한 설정. DB 테이블로 관리하여 코드 수정 없이 신규 기관 추가가 가능하다.

중복 방지 Hash (txn_hash)

계좌ID + 거래일시 + 금액 + 거래처명을 조합한 SHA-256 해시값. 동일한 거래를 여러 번 가져오기 해도 중복 저장을 자동으로 방지한다.

수동 입력

CSV로 수집되지 않는 현금 거래나 기타 지출을 사용자가 직접 입력하는 기능.

Dart Isolate

Dart의 경량 병렬 처리 단위. UI 스레드와 분리된 별도 Isolate에서 CSV 파싱 및 DB 저장 작업을 수행하여 UI 블로킹을 방지한다.

프로필 (Profile)

사용자 2인 각각의 사용자 단위. 각 프로필은 독립적인 계좌 목록, 거래 내역, 설정을 가진다.

통합 뷰 (Combined View)

두 프로필의 계좌와 거래 내역을 하나로 합산하여 보여주는 조회 모드.

03

기능 요구사항 (Functional Requirements)

3.1.1 CSV 파일 가져오기

ID요구사항 내용
FR-CSV-01사용자는 금융기관 앱/웹에서 내보낸 CSV 또는 엑셀(.xlsx) 파일을 앱으로 가져올 수 있어야 한다.
FR-CSV-02파일 가져오기는 파일 탐색기 열기, 드래그 앤 드롭 두 가지 방식을 지원해야 한다.
FR-CSV-03가져오기 시 대상 프로필과 계좌를 선택하거나, 신규 계좌로 등록할 수 있어야 한다.
FR-CSV-04파일 가져오기는 UI 스레드와 분리된 Dart Isolate에서 처리하여 UI 블로킹 없이 동작해야 한다.
FR-CSV-05가져오기 진행 중 진행률(파싱 건수 / 전체 건수)을 프로그레스 UI로 표시해야 한다.
FR-CSV-06가져오기 완료 후 신규 추가 건수, 중복으로 건너뛴 건수, 오류 건수를 요약하여 표시해야 한다.
FR-CSV-07파일 형식이 올바르지 않거나 파싱 오류 발생 시 사용자에게 친화적인 오류 메시지를 안내해야 한다.
FR-CSV-08사용자가 가져오기를 취소할 수 있어야 하며, 취소 시 이미 저장된 데이터는 롤백되어야 한다.

3.1.2 기관별 CSV 파서 프로필

ID요구사항 내용
FR-PARSER-01각 금융기관의 CSV 컬럼 구조, 날짜 형식, 인코딩, 구분자 등을 정의하는 파서 프로필을 DB 테이블로 관리해야 한다.
FR-PARSER-02파서 프로필은 코드 수정 없이 DB에서 추가·수정·비활성화할 수 있어야 한다.
FR-PARSER-03앱은 다음 기관의 파서 프로필을 기본 내장해야 한다: KB국민은행, 신한은행, 우리은행, 하나은행, 농협은행, 카카오뱅크, 토스뱅크, KB국민카드, 신한카드, 삼성카드.
FR-PARSER-04파서 프로필에 없는 기관의 파일 가져오기 시, 컬럼 매핑을 수동으로 설정하는 커스텀 파서 기능을 제공해야 한다.
FR-PARSER-05파싱된 각 거래의 txn_hash(계좌ID + 거래일시 + 금액 + 거래처명의 SHA-256)를 계산하여 중복 저장을 자동으로 방지해야 한다.

3.1.3 계좌 등록 및 관리

ID요구사항 내용
FR-ACC-01사용자는 수동으로 계좌를 등록할 수 있어야 하며, 등록 항목은 프로필, 금융기관, 계좌별명, 계좌 유형(입출금/카드)이다.
FR-ACC-02앱은 최대 2개의 프로필을 지원하며, 각 프로필은 독립적인 계좌 목록과 거래 내역을 가진다.
FR-ACC-03각 프로필에는 이름과 색상(구분용)을 지정할 수 있어야 한다.
FR-ACC-04두 번째 프로필 추가는 첫 번째 프로필 생성 완료 후 설정 화면에서 수행할 수 있어야 한다.
FR-ACC-05계좌별 마지막 CSV 가져오기 일시와 총 거래 건수를 계좌 목록에서 확인할 수 있어야 한다.
FR-ACC-06계좌 삭제 시 해당 계좌의 모든 거래 내역이 함께 삭제되며, 실행 전 확인 절차를 거쳐야 한다.

3.2.1 거래 내역 조회

ID요구사항 내용
FR-TXN-01전체 계좌의 통합 거래 내역과 계좌별 거래 내역을 리스트 형태로 조회할 수 있어야 한다.
FR-TXN-02각 거래 항목에는 거래일시, 거래처명, 입출금 금액, 잔액, 카테고리, 메모 정보를 표시해야 한다.
FR-TXN-03날짜 범위 필터(오늘, 이번 주, 이번 달, 사용자 지정)를 제공해야 한다.
FR-TXN-04금액 범위, 카테고리, 금융기관별 필터링을 지원해야 한다.
FR-TXN-05거래처명 또는 메모를 기준으로 텍스트 검색이 가능해야 한다.
FR-TXN-06거래 내역을 날짜, 금액, 카테고리 기준으로 오름차순/내림차순 정렬할 수 있어야 한다.
FR-TXN-07거래 항목을 선택하면 상세 정보 화면으로 이동하여 카테고리 수정, 메모 입력 등이 가능해야 한다.
FR-TXN-08거래 내역 조회 시 "사용자A / 사용자B / 전체(합산)" 중 선택하여 조회할 수 있어야 한다.
FR-TXN-09거래 항목에 어느 프로필의 계좌에서 발생한 거래인지 시각적으로 구분되어야 한다. (예: 프로필 색상 태그)

3.2.2 카테고리 분류

ID요구사항 내용
FR-CAT-01기본 카테고리: 식비, 교통비, 의류, 의료/건강, 문화/여가, 교육, 공과금, 통신, 금융, 이체, 기타(미분류).
FR-CAT-02CSV 파싱 시 거래처명을 category_rules 테이블의 키워드와 LIKE 매칭하여 카테고리를 자동 분류해야 한다.
FR-CAT-03사용자가 거래 항목의 카테고리를 수동으로 수정할 수 있어야 한다.
FR-CAT-04사용자 정의 카테고리를 추가/수정/삭제할 수 있어야 하며, 카테고리별 색상과 아이콘을 지정할 수 있어야 한다.
FR-CAT-05분류 규칙(category_rules)을 사용자가 직접 추가/수정/삭제할 수 있어야 한다.

3.2.3 잔액 및 자산 현황

ID요구사항 내용
FR-BAL-01대시보드에 연결된 모든 계좌의 합산 총 잔액(최종 CSV 가져오기 기준)을 표시해야 한다.
FR-BAL-02계좌별 잔액(최종 가져오기 기준)과 기관명, 계좌별명을 목록으로 표시해야 한다.
FR-BAL-03이번 달 총 수입, 총 지출, 순 수지(수입-지출)를 대시보드 상단 요약 카드에 표시해야 한다.
FR-BAL-04잔액 및 금액은 원화(₩) 단위로 천 단위 구분자(콤마)와 함께 표시해야 한다.
FR-BAL-05대시보드에서 "개인 뷰 / 통합 뷰"를 전환할 수 있는 토글을 제공해야 한다.
FR-BAL-06개인 뷰에서는 현재 선택된 프로필의 계좌 잔액과 수입/지출만 표시한다.
FR-BAL-07통합 뷰에서는 두 프로필의 잔액 합산과 수입/지출 합산을 표시한다.

3.2.4 지출 리포트

ID요구사항 내용
FR-RPT-01카테고리별 지출 비율을 도넛/원형 차트로 시각화하여 제공해야 한다.
FR-RPT-02월별 수입/지출 추이를 막대 또는 꺾은선 차트로 제공해야 한다.
FR-RPT-03카테고리별 이번 달 지출 합계와 전월 대비 증감률을 표시해야 한다.
FR-RPT-04리포트 기간을 월 단위로 선택하여 과거 데이터를 조회할 수 있어야 한다.
FR-RPT-05리포트는 "개인 뷰 / 통합 뷰" 전환이 가능해야 한다.

ID요구사항 내용

FR-MAN-04수동 입력 거래는 수정 및 삭제가 가능해야 한다.

3.3 데이터 저장 · 보안 · 설정

ID요구사항 내용
FR-DB-01모든 거래 데이터는 외부 서버로 전송하지 않고 사용자 PC 내 SQLite 파일(fortune_garden.db)에 저장해야 한다.
FR-DB-02앱은 사용자 데이터를 각 운영체제(OS)가 지정한 표준 앱 데이터 디렉터리에 저장해야 한다.
FR-DB-03DB 스키마 변경 시 마이그레이션을 자동으로 수행하여 기존 데이터를 보존해야 한다.
FR-DB-04사용자가 설정 화면에서 DB 백업 파일을 내보내기(Export)하고 복원(Import)할 수 있어야 한다.
FR-DB-05앱 설정에서 모든 데이터를 초기화(전체 삭제)하는 기능을 제공해야 하며, 실행 전 확인 절차를 거쳐야 한다.
FR-SEC-01계좌번호 등 민감 정보는 AES-256 이상의 암호화 알고리즘을 사용하여 SQLite DB에 저장해야 한다.
FR-SEC-02암호화 키는 OS의 키체인(Windows Credential Manager)에 저장하여 앱 외부에서 접근할 수 없도록 해야 한다.
FR-SEC-03앱 시작 시 선택적 PIN 잠금 화면을 제공해야 한다(사용자 활성화 옵션).
FR-SEC-04로그에 민감 정보(계좌번호, 금액 등)가 평문으로 출력되지 않아야 한다.
FR-CFG-01앱 테마(라이트/다크/시스템 기본)를 설정할 수 있어야 한다.
FR-CFG-02금융기관별 계좌별명(닉네임)을 설정할 수 있어야 한다.
FR-CFG-03사용자 설정은 SQLite app_settings 테이블에 저장되어 앱 재시작 후에도 유지되어야 한다.
FR-CFG-04각 프로필의 이름, 색상, 기본 뷰(개인/통합) 설정을 변경할 수 있어야 한다.
FR-CFG-05앱 시작 시 기본으로 표시할 뷰(개인 뷰 — 어느 프로필 / 통합 뷰)를 설정할 수 있어야 한다.

4.2 가용성 · 사용성 · 유지보수성 · 보안

ID요구사항 내용
NFR-AVAIL-01네트워크 오프라인 상태에서도 로컬 DB에 저장된 모든 거래 내역 조회, 카테고리 수정, 메모 입력이 가능해야 한다.
NFR-AVAIL-02앱 강제 종료 또는 시스템 재시작 후에도 마지막 가져오기된 데이터가 정상적으로 유지되어야 한다.
NFR-AVAIL-03CSV 가져오기 실패 또는 취소 시 이전에 저장된 데이터가 손실되지 않아야 한다(원자적 트랜잭션 보장).
NFR-AVAIL-04파일 파싱 오류 발생 시 앱이 비정상 종료되지 않고 오류 메시지를 표시하며 정상 동작해야 한다.

NFR-UX-02UI 언어는 한국어를 기본으로 하며, 숫자/날짜는 한국 로케일 형식(YYYY년 MM월 DD일, ₩)을 따른다.
NFR-UX-03데스크탑 환경에 최적화된 키보드 단축키(예: Ctrl+I: 파일 가져오기, Ctrl+F: 검색)를 제공해야 한다.
NFR-UX-04오류 메시지는 기술적 용어를 지양하고 사용자가 이해할 수 있는 친화적 언어로 안내해야 한다.
NFR-UX-05최소 창 크기는 360×600을 지원해야 하며, 창 크기 변경 시 레이아웃이 반응형으로 조정되어야 한다.
NFR-MAINT-01코드는 Clean Architecture 패턴을 준수하여 Presentation, Domain, Data 레이어를 명확히 분리해야 한다.
NFR-MAINT-02새로운 금융기관 CSV 파서 추가 시 기존 코드 변경 없이 DB 파서 프로필 테이블에 항목만 추가하는 방식으로 확장 가능해야 한다.
NFR-MAINT-03카테고리 분류 규칙은 코드에 하드코딩하지 않고 DB 테이블(category_rules)로 관리하여 앱 업데이트 없이 규칙 수정이 가능해야 한다.
NFR-MAINT-04핵심 비즈니스 로직(CSV 파싱, 카테고리 분류, DB 저장)에 대한 단위 테스트 커버리지 70% 이상을 목표로 한다.
NFR-SEC-01앱 실행 중 메모리 덤프로 민감 정보가 노출되지 않도록 민감 데이터의 메모리 잔류 시간을 최소화해야 한다.
NFR-SEC-02민감 데이터는 기본적으로 AES-256 이상의 암호화 알고리즘을 사용하여 저장되어야 한다.

5.1 계층형 아키텍처

Fortune Garden의 시스템은 4개의 주요 계층으로 구성됩니다. 각 계층은 단방향 의존성을 유지하며, Presentation → Domain → Data → External 순으로 호출 흐름이 발생합니다. v1.2에서는 External 계층의 CODEF API가 제거되고 CSV 파일 시스템으로 대체되었습니다.

Presentation (L1)Flutter UI

대시보드, 거래 내역 리스트, 리포트, 계좌 관리, 설정 화면. 상태 관리는 Riverpod 사용.

FlutterRiverpod
Application (L2)UseCase / Service

CSV 파싱 오케스트레이션, 카테고리 자동 분류, 프로필 전환 로직.

5.2 데이터 흐름

CSV 가져오기 흐름: 사용자가 금융기관에서 CSV 내보내기 → Fortune Garden에서 파일 선택(탐색기 또는 드래그 앤 드롭) → 파일 확장자 확인 → 대상 프로필 및 계좌 선택 → Dart Isolate에서 파서 프로필에 따라 CSV 파싱 → 각 행의 txn_hash 계산 → DB 중복 체크 → 신규 거래만 SQLite 저장 → 카테고리 자동 분류(category_rules LIKE 매칭) → Flutter UI에 완료 이벤트 전달 → 화면 갱신

수동 입력 흐름: 사용자가 수동 입력 화면에서 거래 정보 입력(날짜, 금액, 거래처, 카테고리 등) → is_manual=1 플래그와 함께 SQLite transactions 테이블에 직접 저장 → 저장 완료 후 거래 내역 목록 즉시 갱신

07

제약사항 및 가정

7.1 기술적 제약사항

  • Flutter 3.x 이상, Dart 3.x 이상 환경을 사용해야 한다.
  • CSV 파일 파싱은 각 금융기관의 파일 포맷에 의존하므로, 금융기관이 내보내기 형식을 변경할 경우 파서 프로필 업데이트가 필요할 수 있다.
  • SQLite는 단일 사용자 환경에 최적화되어 있어, 동일 DB 파일에 대한 동시 접근 시 잠금 처리 방식을 적용한다.
  • CSV 파일 인코딩은 기관마다 다를 수 있으며(UTF-8, EUC-KR 등), 파서 프로필에 인코딩 정보를 포함하여 처리한다.

7.2 가정 사항

  • 사용자는 각 금융기관 앱 또는 웹에서 거래내역 CSV 파일을 직접 내보내는 방법을 인지하고 있는 것으로 가정한다.
  • 앱은 최대 2인 사용자 환경을 기준으로 설계하며, 동일 PC에서 두 프로필을 관리한다.
  • 초기 개발 범위에서는 은행 입출금 계좌 및 카드 거래에 집중하며, 대출·펀드·보험 데이터는 제외한다.
  • 외부 네트워크 연결 없이도 앱의 모든 핵심 기능(조회, 입력, 리포트)이 동작해야 한다.
